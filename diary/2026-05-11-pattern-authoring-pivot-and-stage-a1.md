# 2026-05-11 — Pattern authoring pivot, section split, and Stage A1 landing

This session split section 27 of DESIGN.md into two files, updated the new section 28 with a substantial design pivot that moves pattern editing out of the inspector and into the Code tab via JavaScript labelled-statement syntax, and landed Stage A1 of a five-stage code sequence that drops the inspector's Band 4 CodeMirror pattern editor in preparation for the Code tab work.

The doc split. Section 27 had grown to cover two distinct concerns: the language layer plus runtime architecture (the architectural split, pattern evaluation flow, modifier vocabulary, signals, tempo, continuous and one-shot firing, the implementation order) which is essentially stable, and the user-facing UI surface (inspector layout, the cursor-as-collider model, visualisation choices) which has been evolving rapidly. Mixing the two in one file was making each edit risky because changes to the UI subsections could not be made cleanly without touching the stable architectural prose. The split landed in one commit as pure reorganisation: section 27 keeps its existing name "Strudel as Pattern Language" and retains the architectural and runtime content; the new section 28 is titled "Pattern Authoring and the Cursor Model" and takes the Inspector integration subsection, the Cursors and marker-layout-and-cursor-as-collider subsection, and the Visualisation subsection. Cross-references resolved cleanly within section 28: the "next subsection" forward references in Inspector integration still point to the Cursors subsection that follows it within the new file. The TOC in DESIGN.md picked up the new section 28 entry. The Visualisation subsection went with section 28 since marker rendering is closely tied to the cursor and marker material in the cursor-as-collider subsection.

The naming was deliberate. Section 27 keeps the original "Strudel as Pattern Language" because what remains is the language and runtime layer. Section 28's name "Pattern Authoring and the Cursor Model" captures both halves of what moved: the authoring surface (inspector layout, Code tab, labelled-statement blocks) and the cursor-as-collider model (cursors as visible-and-collidable presence, mute as the on/off gate, the lockstep between cursor traversal and pattern timing). The unifying theme is how the user creates patterns and how the system visibly enacts them.

The design pivot. The Band 4 CodeMirror surface in the inspector had landed in the previous session as the home for cyclePattern editing. That approach had a clean inspector affordance but pushed the pattern source into a per-object hidden surface that did not coexist with the JavaScript callbacks for the same object. The new shape moves pattern editing into the Code tab (behaviors.js) using JavaScript labelled statements of the form dollar-objectId colon expression, for example dollar spr1 colon note open-paren-quote-c-d-e-f-quote-close-paren. The labelled-statement syntax is valid JavaScript at the parser level (label, colon, statement), which lets the Code tab keep its existing role as a JavaScript file while picking up the new role as the home for pattern source. The strudel REPL's own dollar-colon convention is exactly this labelled-statement form with an empty label; GXW extends the convention by using the label name to carry the object reference.

The Code tab can contain multiple labelled blocks targeting the same object as variants. The composer types dollar-spr1 colon one pattern, then later dollar-spr1 colon another, then later dollar-spr1 colon a third. All three blocks coexist in the file as a kind of pattern library for that object. Cmd-Enter on the cursor's enclosing block promotes that block to the active pattern for its named object by extracting the expression body and writing it to that object's cyclePattern field in scene.json. The previous active block stays in the file as text; the user can switch back to it by parking the cursor in it and Cmd-Entering again.

The scene runtime cache. Per-object cyclePattern in scene.json stores the currently-active pattern expression as plain text without the dollar-objectId prefix. Cmd-Enter writes to this field; the scene runtime reads from it; scene reload reads it directly without re-parsing the Code tab. The Code tab is the editing surface and cyclePattern is the runtime cache. This separation means scene reload does not depend on the Code tab containing any pattern blocks at all, and the user can rearrange or comment out blocks without disturbing what is currently playing.

Inertness at scene-load. Behaviors.js historically runs as JavaScript at scene-load time to register the top-level callback functions into scene.functionMap. The labelled pattern blocks are syntactically valid JavaScript, so under the naive path they would execute at scene-load and call strudel pattern constructors before the engine is loaded, either erroring or producing stray Pattern objects with no live binding. The scene-load path will pre-process behaviors.js through Acorn (already a Code tab dependency for the syntax linter), walk the top-level statements, and split them into two streams: LabeledStatement nodes whose label matches a dollar-prefixed identifier go into a per-object pattern table held as inert source text; everything else gets concatenated back into a JavaScript source string and executed as before for function registration. The labelled blocks remain in the file as the user sees them but are not run at scene-load. Cmd-Enter is the only path that activates them.

Active-block indication. When one of several labelled blocks for an object is the currently-active pattern, that block's tag (the dollar-objectId colon text) renders in GXW's accent green, matching the colour used for editable field borders and checkboxes. Inactive tags render in the default name-token pink from cmTheme.js's contrastOverrides. The matching rule is text-equality: a block is active for its named object when the block's expression body equals the cyclePattern text stored in the scene.json entry. Duplicated blocks with identical expression text both highlight as active, which is an honest reflection of state; the rule could become more sophisticated later but text-equality is simple and robust to file edits. Implementation is a CodeMirror decoration extension that parses the file, walks for labelled statements, and marks the matching block's tag; the extension recomputes on doc change and re-triggers via a CodeMirror StateEffect when scene.json's cyclePattern changes.

Bidirectional navigation between Properties and Code tabs. The pattern row in Band 3 has a Create or Go-to button as the primary inspector-to-Code-tab path: clicking switches to the Code tab and either scrolls to the object's first occurrence (a labelled pattern block or any slot callback function) or scaffolds an empty labelled block at the end of behaviors.js and drops the cursor there. Beyond explicit Go-to clicks, the two tabs follow the canvas selection while active. In the Code tab, when the canvas selection changes to a single object, the editor scrolls to that object's first occurrence in the file. When the canvas selection is multi-object the editor does not scroll because the user is probably working on something that is not focused on one specific pattern, so disturbing the editor's scroll position would be more disruptive than helpful. In Properties, when the user switches from the Code tab to Properties, the Code tab's cursor position is read and the enclosing block's object id is checked against the current canvas selection. If the object is already part of the selection the selection is preserved, since parking the cursor in one of its members' code reads as navigation within that set rather than a request to abandon a deliberate multi-selection. If the object is not in the current selection the selection is replaced with just that one object, since the user's focus has moved to something else and the previous multi-selection is no longer relevant. Tab switches themselves stay explicit user actions; selecting an object on canvas does not auto-switch tabs.

Stage A plan. Five commits land the design pivot in code. Stage A1 drops the inspector's Band 4 editor and leaves cyclePattern temporarily un-editable in the UI while leaving existing values in scene.json running through the runtime unchanged. Stage A2 adds Acorn pre-processing in sceneLoader so behaviors.js with labelled blocks loads cleanly without executing them. Stage A3 adds a Band 3 pattern row with Create or Go-to button as the inspector-to-Code-tab path. Stage A4 adds Cmd-Enter routing in the Code tab that detects labelled blocks and emits setCyclePattern edits. Stage A5 adds the active-tag CodeMirror decoration. Ordering matters: Stage A2 must land before Stage A3 because Stage A3's Create button scaffolds a labelled block in behaviors.js, which triggers a scene reload, which without A2's inertness pre-processing would execute the block at load time and fail. Stages A1 and A2 are independent in principle; A1 first leaves the codebase in a known buildable state before any new pattern-handling logic is added.

Stage A1 landed. The inspector reverts to three bands: Identity, Geometry, Callback slots, followed by the bottom spacer that returned from the pre-Band-4 layout. The patternEditor.js module was removed (git rm); cmTheme.js stays because the Code tab still imports it for syntax theming. The cyclePatternParseError dispatch branch in main.js is gone since the inspector no longer emits that edit kind. The setCyclePattern dispatch in main.js stays because Stage A4's Cmd-Enter routing in the Code tab will re-introduce a path that emits it; the parse-and-log step continues to produce a confirmation line in the GXW console when the path is reactivated. Band 4 CSS (.insp-band-cm, .insp-pattern-header, .insp-pattern-editor, .insp-pattern-error and their children) is gone from main.css. The .inspector-panel rule keeps its flex-column layout, with the rationale updated from supporting Band 4's flex-grow to supporting the simple stack of bands plus bottom spacer. The existing cyclePattern values in scene.json keep firing through the runtime since the simulation path is unchanged; only the UI editing surface is temporarily absent.

Tool quirks encountered. The filesystem MCP's edit_file tool corrupted section 27 during the split when an oldText pattern contained backtick-dollar combinations and the tool matched only a substring of the intended block. Recovery was via write_file to recreate section 27 from scratch with only the kept content. Lesson: for content reorganisation rather than localised text changes, write_file is safer than edit_file even though it requires composing the full file contents. The filesystem MCP also has no delete operation, so removing files (like patternEditor.js this session) requires a terminal git rm before the commit.
