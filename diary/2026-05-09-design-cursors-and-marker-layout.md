# 2026-05-09 — Design discussion: cursors, marker layout, and the cursor-as-collider model

End-of-day design session, no code changes. The result is a new subsection in section 27 ("Cursors, marker layout, and the cursor-as-collider model") that captures several refinements to the inspector and the cycle-firing model. This diary entry walks through the reasoning that led to the subsection so the trade-offs and the pacing of decisions stay readable.

The trigger for the discussion was a use case Chris hadn't surfaced yet. A curve's cyclePattern is supposed to fire audio events when canCycle is on, per section 27 as originally written. Chris's additional use case: even with canCycle off, the cyclePattern should populate the curve with diamond markers along its geometry. The purpose is to let other cursors (from different curves, or free sprites) sweep over those markers and fire collisions, using the mini-notation pattern as a fast authoring tool for trigger layouts. The old beat-points mechanism handled this before the section 27 reshape stripped activeBeats and the per-curve beat-points fields; Chris wants the visual back, but driven by mini-notation rather than the older string-based representation.

The cleanest framing turned out to be: cyclePattern is the source of truth for two interpretations that agree on positions. The audio firing pattern is one interpretation; the visual marker layout is the other. Both come from queryArc against the same parsed Pattern with the same cycleDuration, so the marker positions and the audio event positions are the same fractional-cycle positions. A curve with cyclePattern set always shows markers; whether it also fires audio depends on whether it has a cursor.

That dependence is where the second refinement landed. The earlier design carried a separate canCycle gate as an explicit boolean. Chris proposed deriving cursor presence (and therefore audio firing) from the curve's cursorL or cursorR field being non-zero. The schema field falls out; the user expresses intent by giving the source a cursor extent rather than by toggling a parallel checkbox. The same derivation works for free sprites once they're given cursorL and cursorR fields too. Triggers can never have a cursor since they don't move, so their cursor extent fields are inapplicable and would render greyed in the inspector.

A further consequence Chris made explicit: cursors are the only way sprites can collide with anything. A sprite is conceptually a point object, and point-versus-point collision is geometrically degenerate without some swept area. Requiring a cursor to enable collision dissolves this problem cleanly. The sprite's cursor is a line segment perpendicular to its last motion direction with cursorL and cursorR controlling the extents to either side, so between consecutive simulation ticks it sweeps a quadrilateral that triggers can be tested against. A sprite without a cursor moves around as a passive point that can't hit anything. The same gesture controls visual presence and collision capability, which is what good design looks like; the user authors collision by giving the sprite a cursor, full stop.

The third refinement was the lockstep illusion that falls out of the master clock. The cursor's t advances at one full cycle per cycleDuration, and the pattern's events are scheduled at fractional cycle times multiplied by the same cycleDuration. They share one clock, so the cursor reaches each marker at exactly the moment that marker's audio event fires. The user reads this as cursor-causes-firing because that's the simplest causal story their eye can construct, even though the pattern is firing on its own scheduler and the cursor is independently traversing the geometry. This isn't an accident; it's a property of the master-clock architecture and means we don't have to engineer the lockstep — it just holds.

For stochastic patterns the marker positions change cycle to cycle. The decision was to re-render markers at each cycle wrap by re-querying the pattern. Deterministic patterns produce stable layouts and the re-render has no visible effect; stochastic patterns produce visibly shifting layouts and the user sees the diamonds move. When the curve also has a cursor and is firing audio, the cursor's traversal is in lockstep with whichever positions the current cycle produced, so the visual coincidence holds even for stochastic patterns. Stage 3's master clock already advances cycle phase unconditionally for every source, so we have a clean cycle-wrap moment to hook the marker re-render onto.

A few things were filed as future possibilities rather than acted on. Marker layout for sprites by predicting their trajectory is interesting (the sprite's near-term path could be sampled forward and markers placed along the predicted positions) but gets complicated fast: collisions perturb the prediction, and a marker the sprite was supposed to hit could be invalidated by an intervening wall bounce. Sprite-vs-sprite collision is also deferred; sprite-vs-trigger covers most of the use case and is geometrically simpler. The Band 4 CodeMirror surface for multi-line patterns is deferred indefinitely; the inline single-line cyclePattern field on Band 3 row 1 is sufficient, scrolling within its width if patterns get long. The detailed mechanism for ctx population on a marker-collision-driven beenHit (when a curve has cyclePattern but no cursor and gets hit by another's cursor at marker N) is deferred until collision firing is wired.

The same cursor-presence derivation extends to canHit. A source can only do the hitting via its sweeping cursor, so canHit collapses into the cursor-presence derivation in the same way canCycle did. canBeHit stays a separate gate, since being hittable doesn't require a cursor: anything with a body can be hit by another's cursor. canTick stays a separate gate too, since onTick is purely time-based. canHit's removal is provisional; the field stays for now and the derivation lands when collision firing is wired.

The Band 3 row 1 row label currently reads canCycle. Whether to rename it (selfDrives, fires, playsCycle, hasCursor as candidates) was discussed but not settled. Any name has to read sensibly across all three source kinds, and the row label is a small thing that can ride along whenever it's most convenient. For now the label stays as canCycle and serves as a passive label for the cyclePattern field rather than a gate for any behaviour.

Tomorrow's session opens with the cursor-as-collider work as a probable next stage: removing canCycle from the schema, adding cursorL / cursorR to sprites, parsing cyclePattern as mini-notation, rendering markers along curves, and deriving cursor presence from the cursor extent fields. Or with the simpler audio firing path that calls JS callbacks on cycle wrap without strudel, if the schema changes feel like the wrong thing to land first. Either path leaves the other for the stage after.
