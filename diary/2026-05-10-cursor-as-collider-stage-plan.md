# 2026-05-10 — Cursor-as-collider stage plan

Afternoon session continuing from this morning's design refinement on mute as cursor gate. With the design points settled, we picked the cursor-as-collider work as the next concrete stage and walked through the inspector reorganisation that comes with it. The shape that fell out is bigger than the state file's earlier sketch. Not just the schema and visibility changes the cursor-as-collider model needs, but a related restructuring of Band 1, Band 3, and Band 4 that simplifies the inspector while making the cycle duration field universally accessible. Three commits batched at the end of the sequence will land the work.

Band 1 changes. The user-typed Name field on Band 1 row 2 is dropped from the inspector. The schema field stays in place so existing saved scenes preserve their names without a migration, and a future change can re-surface the field. The system-assigned object ID (kind underscore six hex characters) becomes the only identifier the user sees. The existing greyed Object ID display in Band 1 row 1 already provides ID visibility; the GXSTR-style title bar treatment that shows kind-plus-name for single-selected named objects is deferred to whenever the simplified-naming work lands, which itself is a future cleanup that changes the auto-generated id format to numbers following the object kind. The freed row in Band 1 holds a single-line cycle duration row: a "cycle duration" label, then a small numeric field, then a "beats" suffix. The schema field stays named beatsPerCycle since it semantically represents beats per cycle; the UI label and units suffix describe the same concept from a different angle. The field greys when the selected object is a trigger, since triggers cannot self-fire under the cursor-as-collider model and their cycle counter tracking is internal-only.

Band 3 simplification. The first row holding the canCycle checkbox, the cyclePattern text field, and the deferred Create button is removed entirely. The canCycle gate is gone from the schema since cursor presence is now derived from cursor extents and mute, and the cyclePattern moves to a newly-active Band 4 CodeMirror surface. The second row holding the beatsPerCycle field is also removed since the field moves to Band 1. After this Band 3 has just three rows: hasHit, beenHit, onTick, each with its existing layout of a Can-X gate, function name field, and Create or Go-to button.

Band 4 activation. The CodeMirror region that has been a stub since Stage 2B activates as the home for the cyclePattern field. Always editable, no Here versus Code Tab choice. The Code Location radio drawn inert in Band 4 also goes away in this stage, since Code Tab is no longer an option for the cycle pattern. The cyclePattern stays editable for triggers even though their cycle duration is greyed, because future Tier 5 collision-firing work will fire trigger patterns one-shot on collision using a flourish duration rather than a self-driven cycle. This change un-defers the Band 4 CodeMirror surface that the cursor-as-collider subsection of section 27 had previously deferred in favour of an inline single-line field on Band 3 row 1; that subsection of section 27 will be updated to reflect the new layout as part of the first stage commit.

Schema changes. Removal of canCycle and cyclePatternLocation from the shared CALLBACK_SLOT_FIELDS. Addition of cursorL and cursorR to SPRITE_FIELDS, defaulting to zero so existing sprites do not silently grow cursors on schema migration. Cursor thickness for sprites is deliberately not added at this point; sprite cursor visualisation comes when actual rendering work lands, which is separate from this stage. The sceneEditor's stripObsoleteFields pass picks up an additional sweep to clear canCycle and cyclePatternLocation from existing scenes' stored JSON on next load.

Cursor visibility derivation. Once the schema fields are in place, cursor presence becomes a derived signal of cursorL or cursorR non-zero AND mute unchecked. The inspector greys the cursor extent fields when mute is checked, and the canvas hides the cursor when mute is checked. For curves the cursor rendering already exists, so the change is a gating check at draw time. Sprite cursor rendering on the canvas is deferred to a later piece of work, since adding the field schema and visibility logic does not require drawing sprite cursors visually right now.

Three-stage shape. Stage one bundles the structural and visibility changes: schema removals and additions, all of the Band 1, Band 3, and Band 4 reorg, cursor visibility derivation with mute integration, and the section 27 update. Stage two adds mini-notation parsing of cyclePattern: parse on commit with caching, parse-as-you-type feedback in the CodeMirror editor, hard rejection of unparseable patterns at commit. Stage three adds marker rendering on curves: plain uniform diamonds at queryArc fractional positions along the geometry, re-rendered on each cycle wrap, with no per-marker visual distinction for accent or gain yet.

Each stage produces a working build at its end. After stage one the inspector matches the new model and the simulation gates cursors correctly, but the cyclePattern in the CodeMirror does not yet parse to anything used. After stage two patterns parse and errors surface in the editor, but no audio fires; audio firing is Tier 2 work proper, beyond this stage. After stage three curves with cyclePattern show diamond markers along their geometry, re-rendering on each cycle wrap to handle stochastic patterns. Section 27's open question on ctx population for marker-collision-driven beenHit stays deferred until collision firing is wired in Tier 5.
