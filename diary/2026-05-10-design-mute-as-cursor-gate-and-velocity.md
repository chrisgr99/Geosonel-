# 2026-05-10 — Design refinement: mute as cursor gate, initial velocity inspector field

Morning continuation of yesterday evening's design discussion on cursors and the cursor-as-collider model. Two refinements landed: an overnight insight from Chris about reusing the existing mute field as the cursor gate, and a small but real omission Chris flagged about the inspector having no way to set a sprite's initial velocity. Both captured into section 27 (the mute refinement) and the state file (the velocity field as a Tier 3 follow-up).

The mute refinement. Yesterday's design derived cursor presence from the cursorL or cursorR field being non-zero. The user expresses cursor intent by giving the source a cursor extent; no separate canCycle boolean. Chris's overnight rethink: this loses the ability to toggle the cursor on and off without zeroing the extent fields and losing those settings. The replacement is to keep the extent-based derivation as the user's authoring affordance ("draw a cursor by giving it size") and add the existing mute field as an explicit operational toggle. A source has a visible, firing cursor iff cursorL or cursorR is non-zero AND mute is unchecked. When mute is checked, the cursor extent fields render greyed in the inspector and the cursor disappears from the canvas; the visual matches the absence of audio. Toggling mute back uncovers the same cursor with the same extents as before.

Reusing mute conserves vocabulary. Mute already means "this source produces no sound" everywhere in the inspector. The refinement extends the metaphor: for a source whose audio comes from its own cursor sweeping, no sound implies no cursor visible, since the cursor is the visible cause of the sound. The user's mental model of mute (silence the channel) lines up with the visual behaviour (hide the cursor that was making the noise). The match between audio absence and visual absence is satisfying.

Mute and external collisions. The interpretation question worth deciding: when a muted curve's markers get hit by another cursor, do those collisions fire audio or are they silenced too? Chris settled on interpretation one. Mute silences only the source's own self-firing — the audio that fires when the source's own cursor sweeps over its own markers. External collisions fire normally regardless of mute state, since those collisions are credited to the colliding source's hasHit callback (its own audio output) and to the muted source's beenHit callback (whose own gate canBeHit is independent of mute). This keeps the passive-curve-as-trigger-field use case clean: a user can lay out a curve with cyclePattern, mute the curve so its own cursor stays silent, and the curve's markers remain audible when other cursors sweep across them. The sound is credited to the playing cursor and the curve's beenHit binding rather than to the curve's own muted output.

For triggers, mute is a no-op behaviourally. Triggers cannot self-fire under the cursor-as-collider model since they have no cursor, so there is nothing for mute to silence. The mute field stays in the trigger schema for uniformity with curves and sprites, and a muted trigger renders the same way an unmuted one does.

The initial velocity inspector field. A separate gap Chris flagged that wasn't on the list. Sprites carry vx and vy fields in the schema, but the inspector currently has no surface for editing them; the only way to set a sprite's initial velocity is to hand-edit scene.json. Chris wants velocity X and velocity Y fields in Band 2 alongside the existing position X and position Y row, so the user can author "where the sprite starts and how it starts moving" as one visual unit. The four numeric fields go on a single row to conserve vertical space, with the constraint that the leftmost field's left edge aligns with the leftmost field's left edge in other Band 2 rows so the column structure stays consistent down the band.

Layout for the four-numeric row will need some compression. Band 2's current position row has roughly leftLabel + X field + Y field, which is already wider than most rows; adding two more fields on the same row pushes the band wider than it can comfortably be. Options the inspector implementation will weigh: drop the gap between the X-Y pair and the vx-vy pair so all four numbers sit close together with a single label like "x y vx vy" on the left, or shrink the field widths slightly so all four fit in roughly the current row width, or use a packed representation where the row reads as four numbers with a single shared label. The exact answer depends on how cramped each option looks in practice; the design constraint Chris gave is just that the leftmost field has to align with the column above and below.

Velocity edits during playback work the same way drags and inspector position edits do today: the field commits to the sprite's authored vx and vy, setScene reconciliation detects the authored change against the per-state record, and the runtime vx and vy snap to the new authored value on the spot. So a user can adjust a sprite's velocity mid-playback and see the change take effect immediately rather than waiting for a rewind. This falls out of Stage 3's edit-detection logic without any new code.

Triggers and curves don't have vx and vy fields in the schema. For multi-kind selections that include triggers or curves alongside sprites, the velocity columns render greyed for the kinds that don't apply, the same way Band 1's Hide checkbox handles kind-specific applicability today.

Section 27's "Cursors, marker layout, and the cursor-as-collider model" subsection has been updated with the mute refinement; the velocity field is captured in the state file as a Tier 3 follow-up since it's a UI completion item rather than an architectural decision.
