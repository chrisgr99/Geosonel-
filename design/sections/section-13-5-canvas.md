## Section 13.5 — Canvas

The Canvas is GXW's primary visual surface. It renders the live scene — curves with their cyclePattern markers, cursors as line segments sweeping their extents, triggers as discs at their positions, sprites as filled circles coloured from the pixel underneath, the optional background image — and serves as the gesture surface for direct manipulation through the canvas toolbar (see Section 13 for the toolbar and selection model). This section currently covers the Canvas inspector tab and the pattern-marker visualisation subsection from the section 27 and 28 fold-back. The broader Canvas consolidation pass folding content from sections 13, 21, 22, and 26 is still deferred to a future revision.

### Canvas inspector tab (shipped through Stage 4)

The Canvas inspector tab sits in the right-hand tab strip directly to the right of the Properties tab and ahead of the J tab and the per-.js-file code tabs. Unlike the Properties tab, which is selection-driven and shows fields for whichever scene object is currently selected, the Canvas inspector tab is selection-independent and shows scene-level canvas state: which background image is active, the canvas's pixel dimensions, and a gallery of imported images for one-click reuse. The image-transformation parameters (brightness, glare) still live in the Settings dialog as of Stage 4 and migrate to this tab in Stage 6.

Layout, top to bottom. A top row holding the Load Image button, the Remove Image button, and the Canvas W and Canvas H numeric fields. The Load Image button opens a native file picker; the chosen file is normalized (see Image normalization below) and becomes the score's background image, with a thumbnail joining the gallery automatically. The Remove Image button clears the current background. The Canvas W and H fields are the numeric fields migrated from the canvas toolbar so the toolbar can stay focused on object-creation tools. The two buttons together replaced the Load Background Image and Remove Background Image entries that previously lived in the File menu; both menu entries have retired.

A horizontal separator across the inspector.

The image gallery. Four columns wide, scrollable vertically, with each thumbnail rendered at a fixed 72 by 72 pixels. The grid total width is 312 pixels (four 72-pixel columns plus three 8-pixel gaps), left-aligned within the inspector pane. The fixed pixel sizing means the gallery's footprint stays constant when the inspector / canvas pane divider is dragged, which keeps the drag visibly responsive — a fluid 1fr layout was tried in Stage 3 and produced visible lag during drag because every thumbnail resized on every drag frame. Each thumbnail shows the image's pixel content downsized to 96 by 96 pixels, letterboxed to preserve aspect ratio rather than centre-cropped. The currently-active background's thumbnail carries the same green active-frame the Properties tab uses for active fields. Clicking a thumbnail sets that image as the current background immediately, runs the score's auto-save, and moves the green active-frame to the clicked slot.

Image normalization. Every image entering the system through any path (toolbar Image button, the Canvas tab's Load Image button, drag-and-drop onto the canvas, paste) flows through the same normalization step: decode via createImageBitmap, draw onto a hidden 1000 by 1000 canvas with a hard stretch (no aspect-ratio preservation), and re-encode as JPEG at quality 0.70. The hard stretch is intentional. The image is treated as a varying signal source for music generation rather than a faithful visual representation, and the canvas's own canvasW by canvasH display geometry inversely re-stretches roughly back toward typical input aspect ratios. The deterministic JPEG encode means re-decoding produces the same pixel values on every reload, so signal sampling is stable across sessions. JPEG at quality 0.70 gives a 5-to-10x disk footprint reduction over PNG for photographic content; the bytes inside score bundles and inside the gallery cache are always in this normalized form regardless of source format.

Image identity is content-hash based. The bundle carries an imageContentHash field which is the SHA-256 hex of the normalized 1000 by 1000 JPEG bytes; gallery entries carry the same hash. Score open synchronizes the bundle's image against the gallery: if a matching hash exists, the green active-frame moves to that entry; if not, the bundle's image is registered as a new gallery entry with a freshly-computed thumbnail. Hash-matching survives a bundle being moved between machines, since identical normalized bytes always produce the same hash. Legacy bundles saved before the imageContentHash field was added have their hash computed at load time and assigned to the in-memory bundle (no forced save).

Gallery position is stable. addedAt is immutable after an entry's creation. Clicking a thumbnail, opening a score whose background matches an existing entry, and re-importing the same image are all no-ops for slot ordering — none of them shifts entries between slots. Fresh imports append at the end of the gallery. The result is a curated-library feel where the user can develop positional memory for the images they're working with. Eviction at cap currently drops the oldest entry by addedAt, which technically shifts one slot in the corner case; this gets revisited as part of the Stage 5 redesign described below.

Storage model. The gallery's metadata (entry ids, source paths, base64 thumbnails at 96 by 96 pixels, addedAt timestamps, content hashes) lives in settings.json under Application Support, alongside other app-level settings. Full-resolution copies of each gallery image live in a dedicated imageCache folder under the same Application Support directory as JPEG files at 1000 by 1000 pixels at quality 0.70. The cache copy is what the Canvas loads when a thumbnail is clicked, so the gallery survives the source file being moved or deleted. Thumbnails are generated renderer-side at import time by drawing the normalized image into a hidden 96 by 96 canvas and encoding the result as base64 PNG. Score bundles embed their own copy of the chosen image as today; the cache and the bundle are independent storage locations.

Gallery cap. The current cap is 40 entries on Electron and 12 on the web build. The Electron cap is not yet user-configurable but will be exposed in the Settings dialog as part of the Stage 5 work below, with a range of 5 to 200. The web cap is fixed and not exposed in any UI; the small fixed value matches the web build's general "good enough for casual use, get the desktop app for serious work" stance and keeps the IndexedDB footprint bounded.

Empty state. The empty-state gallery (first launch, no entries yet) shows 20 empty placeholder slots with a neutral background. Load Image or any of the other import paths populate the gallery as entries get added.

Keyboard. Mouse-only by design. The Canvas inspector tab does not participate in tab-key focus traversal of the inspector — thumbnails are clickable but not focusable, and the buttons and fields use their native focus behaviour without a dedicated tab-order pass.

### Stage 5 redesign — two-tier gallery and Pin button (planned, not yet implemented)

The single-gallery model above evolves into a two-tier model in Stage 5, motivated by live performance use. Composers want a curated set of images that travel with the score, separate from a personal working pool that the user accumulates across many scores. The single global gallery doesn't model this naturally — the same list serves both purposes and tends toward awkward compromise. The Stage 5 redesign supersedes an earlier Stage 5 plan (right-click Remove from history plus a max-recents Setting field); the right-click context menu drops from scope entirely.

Two-tier layout. The Canvas tab gains a per-score pinned section above the existing shared gallery, separated by a horizontal divider. The pinned section is a 2-row by 3-column grid of up to 6 slots, each holding an image stored inside the .gxs bundle that travels with the score when shared between users. Empty slots render as placeholders. The shared gallery below is the same global gallery as today, but its semantics return to recency-bump: new imports prepend, clicks on shared thumbnails promote to front. The user's positional memory lives in the pinned section above where positions stay stable; the shared section is allowed to reflow because important images are pinned above.

Layout dimensions. Both sections render at three columns wide, with thumbnails sized to fill the available column width minus a small inter-thumbnail gutter. The gallery total width matches the inspector's widest row (the Starting State row in Band 2 — label, X, Y, vX, vY fields with their inline letter prefixes), which is the natural pane-width floor the inspector pane already commits to via main.js's body-divider sizing. The gain over Stage 4's 4-column 72-pixel layout is per-thumbnail legibility — thumbnails roughly double in display size — which matters for accessibility. The cached thumbnail bitmap upgrades from 96 by 96 pixels to 240 by 240 pixels so thumbnails stay crisp on retina displays at the larger display size; cache footprint at the new shared-gallery default cap of 48 entries lands at roughly 1.5 MB, well within reasonable bounds for app-level settings storage.

Storage. The .gxs bundle gains a pinned subfolder holding up to 6 JPEGs named by their content hash. The .gxw-meta.json sidecar gains a pinnedSlots field: an array of 6 entries each either a hash string referencing a file in the pinned subfolder or null for an empty slot. Backward compatible: bundles without the field load with all 6 slots empty. The current-background storage (imageName plus imageContentHash) is unchanged and remains independent of pinnedSlots — a current background may or may not be one of the pinned images.

Pin button. The Canvas tab top row gains a Pin button alongside Load Image, Remove Image, W, H. Clicking it copies the current background's bytes into the bundle's pinned subfolder and writes the hash into the first empty pinnedSlots entry. The button is enabled when a current background exists, at least one slot is empty, and the current background's hash isn't already in pinnedSlots; disabled in any of those negative cases. A click on a disabled button surfaces a brief message in the messages area explaining why ("All pinned slots are full, drag one off to make room", or "This image is already pinned to slot N", or "No current background to pin"). The Pin button is the only way to add to the pinned section; cross-section drags are out of the design entirely.

Interactions summary. Click any thumbnail (pinned or shared) sets it as the current background. Click on a pinned slot also adds the image to the shared gallery if not already present, so the recipient of a shared score builds up their own shared gallery only from images they actually use. Click on a shared slot promotes the entry to front in shared (recency-bump for shared only). Click the Pin button to add the current background to the next empty pinned slot. Drag a thumbnail within a section to reorder it within that section. Drag a thumbnail off the edge of the gallery to remove it: removing from shared deletes the entry and unlinks its cache file; unpinning from a pinned slot clears the slot and removes the pinned-subfolder file if no other slot references the same hash. Drop outside the gallery triggers a vanish animation (scale to roughly 1.4x, fade to zero opacity over about 350ms) before the removal completes. Click-versus-drag is distinguished by a small motion threshold of about 4 pixels.

Score open is passive in Stage 5. When a score loads, the bundle's pinned slots populate the pinned section on the Canvas tab. The current background renders on the canvas as today. The shared gallery's order and contents are not modified by score-open — opening a score never silently mutates the user's working pool. The green active-frame is set on whichever section's entry matches the bundle's imageContentHash: a pinned slot if pinned there, otherwise the corresponding shared entry if it happens to be in shared, otherwise no active frame at all.

Settings and eviction. The max-recents-count Setting field gets added to the Settings dialog as part of Stage 5 (Electron only, 5 to 200 range, default 48, with immediate eviction on decrease). The default lands at a multiple of three so the gallery's bottom row is always full at default settings; users picking custom values can land on any integer in range. The web build's fixed cap moves from 12 to 18 entries (six rows of three). Eviction policy in the shared gallery shifts to "evict the last entry in array order" (least valued by user-controlled position) when a new import arrives, rather than the current Stage 4 "oldest by addedAt" rule. The pinned section has a fixed cap of 6 and is not exposed in Settings.

Phasing. Three commits. First commit lands the storage model change, the two-tier visual layout with empty placeholders, and the Pin button with its enable/disable logic. Second commit adds the drag interactions (reorder within sections, drag-off-edge to remove or unpin, vanish animation). Third commit handles the max-recents Setting field and the eviction policy revisit.

### Stage 6 work — image transformations (planned, not yet implemented)

The image-brightness and image-glare controls migrate from the Settings dialog to this tab, sitting below the gallery section behind a second horizontal separator. A new uniform-brightness slider sits above them. The image-transform pipeline updates so uniform-brightness applies first as overall exposure and the perceptual brightness reduction described in Section 26 applies afterward as highlight rolloff. The natural authoring order is the new uniform field first (overall exposure), then the perceptual reduction (rein in highlights), which is also the order the controls sit in the tab. Chris has flagged additional control changes in mind that will surface when Stage 6 starts.

Settings dialog disposition. The image-brightness and image-glare controls migrating to this tab leave the Settings dialog with the max-recents-count option from Stage 5 plus whatever other settings already exist there. The dialog stays in place rather than retiring.

### Out of scope for the Canvas inspector tab

Grid display, snap-to-grid, and any other canvas-affordance controls remain unimplemented and will land in a separate commit unrelated to the Canvas inspector tab work.

### Pattern-marker visualisation

A curve's cyclePattern renders as diamond markers along the curve at the events' fractional positions within the cycle. The marker layout for an arbitrary mini-notation pattern is not as simple as evenly-spaced beats: positions follow whatever the parsed Pattern's queryArc returns for each cycle, and that varies with modifiers like subdivision, weighted slots, and within-cycle parallel layers.

Two approaches to marker visualisation are under consideration. Static glyphs at edit time would query the pattern for one cycle when the composer commits a pattern edit (Cmd-Enter on a labelled block) and place visual markers on the curve at the resulting fractional positions. This works cleanly for deterministic patterns and gives the composer a stable visual reference between edits. It breaks down for stochastic patterns (using degradeBy, choose, scramble, or rand/irand signals) where each cycle's events differ.

After-the-fact breadcrumbs would drop a fading marker at each event's actual played position when the audio plays, leaving a visible trail of recent events. This works for any pattern, deterministic or stochastic. It does not pre-show where the next firing will be, but for a tool where the composer is composing in real time and listening, the retrospective view is often more useful than the predictive one.

A hybrid is plausible: static glyphs at edit time for deterministic patterns, after-the-fact breadcrumbs for stochastic ones, with detection of which mode applies happening at parse time (a quick walk of the pattern's AST identifies stochastic modifiers). The final decision on which mode (or hybrid) ships is deferred until the rest of the visualisation surface gets focused attention during the Canvas consolidation pass.
