## Section 13.5 — Canvas

The Canvas is GXW's primary visual surface. It renders the live scene — curves with their cyclePattern markers, cursors as line segments sweeping their extents, triggers as discs at their positions, sprites as filled circles coloured from the pixel underneath, the optional background image — and serves as the gesture surface for direct manipulation through the canvas toolbar (see Section 13 for the toolbar and selection model). This section currently covers the Canvas inspector tab and the pattern-marker visualisation subsection from the section 27 and 28 fold-back. The broader Canvas consolidation pass folding content from sections 13, 21, 22, and 26 is still deferred to a future revision.

### Canvas inspector tab

The Canvas inspector tab sits in the right-hand tab strip directly to the right of the Properties tab and ahead of the J tab and the per-.js-file code tabs. Unlike the Properties tab, which is selection-driven and shows fields for whichever scene object is currently selected, the Canvas inspector tab is selection-independent and shows scene-level canvas state: which background image is active, the canvas's pixel dimensions, the image-transformation parameters that today live in the Settings dialog, and a gallery of recently-imported background images for one-click reuse.

Layout, top to bottom. A top row holding the Load Image button, the Remove Image button, and the Canvas W and Canvas H numeric fields. The Load Image button opens a native macOS file picker; the chosen file becomes the score's background image and joins the gallery's recents. The Remove Image button clears the current background without affecting the gallery. The Canvas W and H fields are the same numeric fields that today live in the canvas toolbar, migrated here so the toolbar can stay focused on object-creation tools. The two buttons together replace the Load Background Image and Remove Background Image entries that today live in the File menu; both menu entries retire when this tab ships.

A horizontal separator across the inspector.

The recent-image gallery. Four columns wide, scrollable vertically, with roughly five rows of thumbnails (twenty entries) visible at the default inspector width. Each thumbnail shows the image's pixel content downsized to ninety-six pixels square, letterboxed to preserve aspect ratio rather than centre-cropped. The currently-active background's thumbnail carries the same green active-frame the Properties tab uses for active fields. Clicking a thumbnail sets that image as the current background immediately, promotes it to slot one of the gallery, and runs the score's normal auto-save pipeline so the change persists. The empty-state gallery (first launch, no recents yet) shows twenty empty slots with a neutral background; Load Image is the only path out of the empty state.

Right-clicking a populated thumbnail opens a small renderer-styled context menu with a single item, "Remove from history", which deletes that entry from the gallery's settings.json record and removes its full-resolution cache file. The source file on the user's disk is untouched. Removing the currently-active background's entry clears the green active-frame in the gallery but doesn't change the background on the canvas; the score's own embedded copy of the image stays intact, so the background just becomes "no longer in the gallery's recents". Right-clicking an empty gallery slot does nothing. The context menu surface is a renderer-side positioned element rather than the native Electron menu, so it inherits the inspector's visual styling and the existing accessibility-zoom behaviour.

A horizontal separator across the inspector.

The image-transformation controls migrated from the Settings dialog: the existing image-brightness and image-glare sliders. Below those, a new uniform-brightness slider that multiplies every pixel value evenly across the image, distinct from the perceptual brightness reduction described in Section 26 which targets bright regions specifically. The natural authoring order is the new uniform field first (overall exposure), then the perceptual reduction (rein in highlights), which is also the order the controls sit in the tab.

Storage model. The gallery's metadata (entry ids, source paths, embedded base64 thumbnails at ninety-six pixels square) lives in settings.json under Application Support, alongside the rest of the app-level settings. Full-resolution copies of each gallery image live in a dedicated cache folder under the same Application Support directory; the cache copy is what the Canvas loads when a thumbnail is clicked, so the gallery survives the source file being moved or deleted. Thumbnails are generated renderer-side at import time by drawing the loaded ImageBitmap into a hidden 96×96 canvas and encoding the result as base64 PNG, keeping the native-module surface unchanged. Score bundles continue to embed their own copy of the chosen image as they do today; the cache and the bundle are independent.

Recency model. Opening a score whose background isn't already in the gallery treats that image as a fresh recent use and adds it at slot one, pushing the oldest entry off the end if the gallery is full. Opening a score whose background is already in the gallery promotes the existing entry to slot one without duplicating. Importing or re-importing an image via Load Image promotes its entry to slot one likewise. The recency model is consistent across all three entry points (Open Score, Load Image, gallery-thumbnail click).

Gallery size. The maximum number of recent images stored is configurable in the Settings dialog, defaulting to forty, with a reasonable upper limit of two hundred. Reducing the limit enforces immediately: entries beyond the new cap drop from settings.json and their cached full-resolution files are deleted in the same write. Increasing the limit takes effect for future imports; previously-evicted entries don't return.

Settings dialog disposition. The image-brightness and image-glare controls migrating to this tab leave the Settings dialog with the new max-recents-count option plus whatever other settings already exist there. The dialog stays in place rather than retiring.

Keyboard. Mouse-only by design. The Canvas inspector tab does not participate in tab-key focus traversal of the inspector — thumbnails are clickable but not focusable, and the buttons and fields use their native focus behaviour without a dedicated tab-order pass.

Deferred. Grid display, snap-to-grid, and any other canvas-affordance controls remain unimplemented and will land in a separate commit.

### Pattern-marker visualisation

A curve's cyclePattern renders as diamond markers along the curve at the events' fractional positions within the cycle. The marker layout for an arbitrary mini-notation pattern is not as simple as evenly-spaced beats: positions follow whatever the parsed Pattern's queryArc returns for each cycle, and that varies with modifiers like subdivision, weighted slots, and within-cycle parallel layers.

Two approaches to marker visualisation are under consideration. Static glyphs at edit time would query the pattern for one cycle when the composer commits a pattern edit (Cmd-Enter on a labelled block) and place visual markers on the curve at the resulting fractional positions. This works cleanly for deterministic patterns and gives the composer a stable visual reference between edits. It breaks down for stochastic patterns (using degradeBy, choose, scramble, or rand/irand signals) where each cycle's events differ.

After-the-fact breadcrumbs would drop a fading marker at each event's actual played position when the audio plays, leaving a visible trail of recent events. This works for any pattern, deterministic or stochastic. It does not pre-show where the next firing will be, but for a tool where the composer is composing in real time and listening, the retrospective view is often more useful than the predictive one.

A hybrid is plausible: static glyphs at edit time for deterministic patterns, after-the-fact breadcrumbs for stochastic ones, with detection of which mode applies happening at parse time (a quick walk of the pattern's AST identifies stochastic modifiers). The final decision on which mode (or hybrid) ships is deferred until the rest of the visualisation surface gets focused attention during the Canvas consolidation pass.
