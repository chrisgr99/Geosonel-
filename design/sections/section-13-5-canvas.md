## Section 13.5 — Canvas

The Canvas is GXW's primary visual surface. It renders the live scene — curves with their cyclePattern markers, cursors as line segments sweeping their extents, triggers as discs at their positions, sprites as filled circles coloured from the pixel underneath, the optional background image — and serves as the gesture surface for direct manipulation through the canvas toolbar (see Section 13 for the toolbar and selection model). This section currently covers the Canvas inspector tab and the pattern-marker visualisation subsection from the section 27 and 28 fold-back. The broader Canvas consolidation pass folding content from sections 13, 21, 22, and 26 is still deferred to a future revision.

### Canvas inspector tab (shipped through Stage 4)

The Canvas inspector tab sits in the right-hand tab strip directly to the right of the Properties tab and ahead of the J tab and the per-.js-file code tabs. Unlike the Properties tab, which is selection-driven and shows fields for whichever scene object is currently selected, the Canvas inspector tab is selection-independent and shows scene-level canvas state: which background image is active, the canvas's pixel dimensions, and a gallery of imported images for one-click reuse. The image-transformation parameters in the Settings dialog (brightness reduction's blur radius, threshold, and max attenuation; see Section 26) remain there and are not affected by the Stage 6 work below — they describe a perceptual highlight-rolloff transform that applies to every score, separate from the per-score display brightness control that Stage 6 adds.

Layout, top to bottom. A top row holding the Load Image button, the Remove Image button, and the Canvas W and Canvas H numeric fields. The Load Image button opens a native file picker; the chosen file is normalized (see Image normalization below) and becomes the score's background image, with a thumbnail joining the gallery automatically. The Remove Image button clears the current background. The Canvas W and H fields are the numeric fields migrated from the canvas toolbar so the toolbar can stay focused on object-creation tools. The two buttons together replaced the Load Background Image and Remove Background Image entries that previously lived in the File menu; both menu entries have retired.

A horizontal separator across the inspector.

The image gallery. Four columns wide, scrollable vertically, with each thumbnail rendered at a fixed 72 by 72 pixels. The grid total width is 312 pixels (four 72-pixel columns plus three 8-pixel gaps), left-aligned within the inspector pane. The fixed pixel sizing means the gallery's footprint stays constant when the inspector / canvas pane divider is dragged, which keeps the drag visibly responsive — a fluid 1fr layout was tried in Stage 3 and produced visible lag during drag because every thumbnail resized on every drag frame. Each thumbnail shows the image's pixel content downsized to 96 by 96 pixels, letterboxed to preserve aspect ratio rather than centre-cropped. The currently-active background's thumbnail carries the same green active-frame the Properties tab uses for active fields. Clicking a thumbnail sets that image as the current background immediately, runs the score's auto-save, and moves the green active-frame to the clicked slot.

Image normalization. Every image entering the system through any path (toolbar Image button, the Canvas tab's Load Image button, drag-and-drop onto the canvas, paste) flows through the same normalization step: decode via createImageBitmap, draw onto a hidden 1000 by 1000 canvas with a hard stretch (no aspect-ratio preservation), and re-encode as JPEG at quality 0.70. The hard stretch is intentional. The image is treated as a varying signal source for music generation rather than a faithful visual representation, and the canvas's own canvasW by canvasH display geometry inversely re-stretches roughly back toward typical input aspect ratios. The deterministic JPEG encode means re-decoding produces the same pixel values on every reload, so signal sampling is stable across sessions. JPEG at quality 0.70 gives a 5-to-10x disk footprint reduction over PNG for photographic content; the bytes inside score bundles and inside the gallery cache are always in this normalized form regardless of source format.

Image identity is content-hash based. The bundle carries an imageContentHash field which is the SHA-256 hex of the normalized 1000 by 1000 JPEG bytes; gallery entries carry the same hash. Score open synchronizes the bundle's image against the gallery: if a matching hash exists, the green active-frame moves to that entry; if not, the bundle's image is registered as a new gallery entry with a freshly-computed thumbnail. Hash-matching survives a bundle being moved between machines, since identical normalized bytes always produce the same hash. Legacy bundles saved before the imageContentHash field was added have their hash computed at load time and assigned to the in-memory bundle (no forced save).

Gallery position is stable. addedAt is immutable after an entry's creation. Clicking a thumbnail, opening a score whose background matches an existing entry, and re-importing the same image are all no-ops for slot ordering — none of them shifts entries between slots. Fresh imports append at the end of the gallery. The result is a curated-library feel where the user can develop positional memory for the images they're working with. Eviction at cap currently drops the oldest entry by addedAt, which technically shifts one slot in the corner case; this gets revisited as part of the Stage 5 redesign described below.

Storage model. The gallery's metadata (entry ids, source paths, base64 thumbnails at 96 by 96 pixels, addedAt timestamps, content hashes) lives in settings.json under Application Support, alongside other app-level settings. Full-resolution copies of each gallery image live in a dedicated imageCache folder under the same Application Support directory as JPEG files at 1000 by 1000 pixels at quality 0.70. The cache copy is what the Canvas loads when a thumbnail is clicked, so the gallery survives the source file being moved or deleted. Thumbnails are generated renderer-side at import time by drawing the normalized image into a hidden 96 by 96 canvas and encoding the result as base64 PNG. Score bundles embed their own copy of the chosen image as today; the cache and the bundle are independent storage locations.

Gallery cap. The current cap is 40 entries on Electron and 24 on the web build. The Electron cap is configurable via the gxw:gallery-set-max-count IPC handler with a 5–200 range; no UI surfaces this. An earlier Stage 5 plan to expose the cap in the Settings dialog was dropped in the third commit. The web cap is fixed and not exposed in any UI; the small fixed value matches the web build's general "good enough for casual use, get the desktop app for serious work" stance and keeps the IndexedDB footprint bounded. The web cap was bumped from 12 to 24 in the third commit so the web grid shows eight rows of three rather than four.

Empty state. The empty-state gallery (first launch, no entries yet) shows 20 empty placeholder slots with a neutral background. Load Image or any of the other import paths populate the gallery as entries get added.

Keyboard. Mouse-only by design. The Canvas inspector tab does not participate in tab-key focus traversal of the inspector — thumbnails are clickable but not focusable, and the buttons and fields use their native focus behaviour without a dedicated tab-order pass.

### Stage 5 redesign — two-tier gallery with drag-to-pin (first commit shipped, second commit in progress)

The single-gallery model above evolves into a two-tier model in Stage 5, motivated by live performance use. Composers want a curated set of images that travel with the score, separate from a personal working pool that the user accumulates across many scores. The single global gallery doesn't model this naturally — the same list serves both purposes and tends toward awkward compromise. The Stage 5 redesign supersedes an earlier Stage 5 plan (right-click Remove from history plus a max-recents Setting field); the right-click context menu drops from scope entirely. Within Stage 5 itself, an initial design featured a Pin button on the top row that pinned the current background to the first empty slot. The first commit shipped that button, then the design evolved (see below) toward a drag-from-shared-to-pinned gesture that replaces the Pin button entirely. The button is removed in the second commit.

Two-tier layout. The Canvas tab gains a per-score pinned section above the existing shared gallery, separated by a horizontal divider. The pinned section is a 2-row by 3-column grid of up to 6 slots, each holding an image stored inside the .gxs bundle that travels with the score when shared between users. Empty slots render as placeholders. The shared gallery below is the same global gallery as today, but its semantics return to recency-bump: new imports prepend, clicks on shared thumbnails promote to front. The user's positional memory lives in the pinned section above where positions stay stable; the shared section is allowed to reflow because important images are pinned above.

Layout dimensions. Both sections render at three columns wide, with thumbnails sized to fill the available column width minus a small inter-thumbnail gutter. Thumbnails are 3:2 (wider than tall) rather than square, matching the typical canvas aspect and the typical landscape imported image; combined with the no-letterboxing pipeline rule (see below), this means the dominant case of "landscape source on landscape canvas" needs no letterboxing in the thumbnail at all. The gallery total width matches the inspector's widest row (the Starting State row in Band 2 — label, X, Y, vX, vY fields with their inline letter prefixes), which is the natural pane-width floor the inspector pane already commits to via main.js's body-divider sizing. Thumbnail bitmap size is 360 by 240 pixels for retina sharpness at the typical display width of 100 to 150 CSS pixels.

No letterboxing anywhere. The image pipeline never letterboxes — not in the cached thumbnail bitmap, not in the display slot, not on the canvas itself. imageNormalize.js hard-stretches every imported image to 1000 by 1000 square JPEG bytes (the image is a signal source rather than a faithful representation), thumbnailGen.js hard-stretches the same bytes to 360 by 240, and the display slot uses object-fit: fill to stretch any thumbnail PNG into the 3:2 slot regardless of its stored aspect. Aspect distortion is the expected and documented behaviour throughout. This rule applies to both the gallery and the canvas, and to both newly-imported and legacy entries.

Storage. The .gxs bundle gains a pinned subfolder holding up to 6 JPEGs named by their content hash. The .gxw-meta.json sidecar gains a pinnedSlots field: an array of 6 entries each either a hash string referencing a file in the pinned subfolder or null for an empty slot. Backward compatible: bundles without the field load with all 6 slots empty. The current-background storage (imageName plus imageContentHash) is unchanged and remains independent of pinnedSlots — a current background may or may not be one of the pinned images.

Pinning gesture (drag-from-shared-to-pinned). The user drags a thumbnail from the shared gallery up into a specific slot in the pinned section. This single gesture chooses the slot and (if the slot is already occupied) which image to replace, in one step. The original Pin button is removed entirely because the drag gesture supersedes it. Cross-section drags were explicitly excluded by the initial Stage 5 design ("drags only operate within a section or off the edge"); this redesign puts one cross-section path back in — shared → pinned for pinning — and leaves the rest of the original rule in place (no shared → shared cross-section moves, no pinned → shared cross-section moves; the within-section reorder and drag-off-edge rules continue to apply).

Drop semantics on a pinned slot fall into two cases:

If the dragged image is not currently in any pinned slot (the common case), the drop replaces whatever was at the target slot. The previous occupant (if any) is unpinned; its bytes are garbage collected from the bundle's pinnedBytes map and its file removed from the pinned subfolder if no other slot references the same hash. Empty target slots simply receive the new pin with no replacement step.

If the dragged image is currently in another pinned slot (the shared thumb of an already-pinned image), the drop is a reorder. The slot positions splice: remove the image from its source pinned slot, then insert it at the target pinned slot, with other slots between source and target shifting to fill the gap. The pinned slot count stays at 6, and the target slot's previous occupant is not replaced — it shifts toward the source position rather than being unpinned. This matches array splice semantics (the simplest mental model for "move and shift").

Click-versus-drag is distinguished by a small motion threshold of about 4 pixels (same pattern as canvas.js).

Interactions summary. Click any thumbnail (pinned or shared) sets it as the current background. Click on a pinned slot also adds the image to the shared gallery if not already present, so the recipient of a shared score builds up their own shared gallery only from images they actually use. Click on a shared slot promotes the entry to front in shared (recency-bump for shared only). Drag a thumbnail (shared or pinned) onto a pinned slot to pin or reorder via the drop semantics above; the pinned-to-pinned case is the natural reorder gesture and uses the same splice path as a shared drag of an already-pinned image. Drag any thumbnail outside the canvas inspector pane to remove: a shared drag removes the entry from the shared gallery (metadata plus cache file); a pinned drag unpins the slot (clears the slot and removes the pinned-subfolder file if no other slot references the same hash). Drop outside the pane triggers a vanish animation (scale to roughly 1.4x, fade to zero opacity over about 350ms) before the removal completes; during the drag the floating preview's border switches from green (drop-on-pinned) to red (will-remove) so the outcome is telegraphed before release. Drop inside the pane but not on a pinned slot is a silent cancel. The shared section does not support within-section reorder — ordering there comes from the recency-bump model, with clicks and imports promoting entries to the front.

Score open is passive in Stage 5. When a score loads, the bundle's pinned slots populate the pinned section on the Canvas tab. The current background renders on the canvas as today. The shared gallery's order and contents are not modified by score-open — opening a score never silently mutates the user's working pool. The green active-frame is set on whichever section's entry matches the bundle's imageContentHash: a pinned slot if pinned there, otherwise the corresponding shared entry if it happens to be in shared, otherwise no active frame at all.

Cap and eviction. The web build's shared-gallery cap moves from 12 to 24 in the Stage 5 third commit; the Electron build stays at its current 40-default, 5–200-configurable behaviour through the gxw:gallery-set-max-count IPC handler. An earlier plan to add a max-recents Setting field to the Settings dialog was dropped, so the Electron cap remains adjustable only by direct IPC call or settings.json edit. Eviction policy in the shared gallery is the Stage 4 "oldest entry by addedAt" rule, which under Stage 5's recency-bump model is effectively least-recently-used since touch, click-promote, and re-import all stamp addedAt to now. The rule fits the two-tier model naturally: pinning is the explicit curate-and-keep gesture, the shared section is the working pool, and a working pool wants "use it to keep it, ignore it to lose it" semantics. An alternative "evict the last entry in array order" was considered separately and rejected — under Stage 5's newest-first addedAt sort and the absence of within-shared drag-to-reorder, that rule produces behaviour identical to "oldest by addedAt" and offers no distinct affordance. A direct "mark for eviction" gesture on shared entries was also considered and rejected; pinning already provides the explicit-keep gesture from the opposite side, and a separate priority signal on shared would duplicate it while implying per-item curation that the design doesn't otherwise express. The pinned section has a fixed cap of 6 and is not exposed in Settings either.

Phasing. Three commits. First commit (shipped) lands the storage model change, the two-tier visual layout, the 3:2 thumbnail aspect with the no-letterboxing rule, and the recency-bump shared gallery. It also shipped the Pin button as originally designed; the button is removed in the second commit. Second commit (shipped) lands the unified drag mechanics: drag from shared or pinned onto a pinned slot pins or reorders, drag outside the canvas inspector pane removes (shared) or unpins (pinned) with the vanish animation. The Pin button and its supporting machinery are removed as part of the same commit. Third commit (shipped) bumps the web build's shared-gallery cap from 12 to 24 entries and leaves the Electron build alone; the previously-planned Settings dialog field was dropped, and the eviction-policy revisit ultimately confirmed the existing Stage 4 rule. See the "Cap and eviction" paragraph above for the full scope rationale.

### Stage 6 — per-score canvas brightness (shipped)

The top of the Canvas inspector tab gains a Brightness slider that dims the displayed background image so the curves, sprites, and triggers drawn over it remain easy to see when the image is busy or high-contrast. The control is purely a display concession: image-derived signals on the music side (pxLt, the OKLCh buffer, anything sampling pixel values for triggers and sprites) continue to read from the unmodified source bitmap regardless of the slider's position, so dimming the image for legibility never changes how the score sounds.

The slider is a native HTML range input with min 0, max 100, step 1, and default 100 (no change). At 100 the displayed image is identical to the source; lowering the value applies a multiplicative globalAlpha at the canvas's drawImage call so the image fades toward the canvas-region background colour. Reaching 0 makes the image effectively invisible. The implementation lives in canvas.js's _drawImage, which saves the current globalAlpha, multiplies it by displayBrightness divided by 100, draws the image, and restores the previous alpha. No bitmap re-derivation runs on slider changes, so the control is cheap to drag continuously.

Layout. The slider sits on its own row between the top button row and the horizontal separator that divides the image controls from the pinned section. The row's label "Brightness" reuses the same .canvas-insp-section-caption style as the "Image:" and "Size:" captions above so the row reads as a sibling of the top row's labelled groups. The slider extends to the right edge of the inspector's natural floor width (the same width the section divider below and the gallery grids beneath that use, set by the CSS variable --canvas-insp-gallery-width that's measured from the Properties inspector's Starting State row at mount time).

The top row itself gains a "Size:" sub-caption inserted between the Clear button and the W field. The sub-caption uses the same .canvas-insp-section-caption style as the row-opening "Image:" caption so the row reads as one strip with two labelled groups (image controls on the left, size fields on the right) rather than as a flat sequence of widgets. The W and H field font-size bumps from 11pt to 13pt with line-height 1 so the numeric content reads at the same visual weight as the captions and is comfortable to scan; the .toolbar-canvas-field class that styles these fields is only used in the Canvas inspector now, so the bump has no effect elsewhere in the UI.

Persistence. The slider value lives on the bundle as a single per-score displayBrightness field, persisted alongside imageName and imageContentHash in the score's record. On the Electron disk backend the field travels in the .gxw-meta.json sidecar so a freshly-loaded score paints at its saved brightness without an explicit save step; on the IDB web backend the field is inline in the score record like the other metadata. Older score records that predate the field load with the default 100, so opening a pre-Stage-6 score looks identical to its previous state until the user explicitly adjusts the slider. The clamp to 0..100 is applied at load time defensively so an out-of-range value edited externally can't push the renderer into an invalid state.

Wiring. The user's slider drag emits the new integer value on every input event; main.js routes the value to both the bundle (bundle.setDisplayBrightness marks the bundle dirty on any actual change) and the canvas (canvas.setDisplayBrightness schedules a redraw). Both setters short-circuit on same-value pushes so a slider that holds at one position doesn't churn through redundant dirty-flag toggles or redraws. On score open, switchToBundle, onExternalChange, and the initial reconcile-with-disk block all sync the canvas's mirror and the inspector's slider position to the freshly-loaded bundle's value so the rendering and the UI stay coherent across every load path. The canvas's value is also synced once at the top of main() so a score loaded with the image visible paints at the right brightness from frame zero rather than briefly flashing at 100 before the inspector wiring block runs.

Settings dialog disposition. The existing Settings dialog brightness-reduction controls (imageDimBlurRadius, imageDimThreshold, imageDimMaxAttenuation, imageDimBypass) are unaffected. Those drive a perceptual highlight-rolloff transform via the Gastál–Oliveira domain-transform filter described in Section 26, which is a heavier per-bitmap derivation that applies to every score and is calibrated for users sensitive to bright pixels. The Stage 6 Brightness slider is a cheap per-score draw-time alpha that solves a different problem (legibility of overlays on a particular image). The two systems compose: the bitmap displayed by _drawImage is already the post-rolloff bitmap when the dim transform is active, and the slider's alpha applies on top of that draw call.

### Out of scope for the Canvas inspector tab

Grid display, snap-to-grid, and any other canvas-affordance controls remain unimplemented and will land in a separate commit unrelated to the Canvas inspector tab work.

### Pattern-marker visualisation

A curve's cyclePattern renders as diamond markers along the curve at the events' fractional positions within the cycle. The marker layout for an arbitrary mini-notation pattern is not as simple as evenly-spaced beats: positions follow whatever the parsed Pattern's queryArc returns for each cycle, and that varies with modifiers like subdivision, weighted slots, and within-cycle parallel layers.

Two approaches to marker visualisation are under consideration. Static glyphs at edit time would query the pattern for one cycle when the composer commits a pattern edit (Cmd-Enter on a labelled block) and place visual markers on the curve at the resulting fractional positions. This works cleanly for deterministic patterns and gives the composer a stable visual reference between edits. It breaks down for stochastic patterns (using degradeBy, choose, scramble, or rand/irand signals) where each cycle's events differ.

After-the-fact breadcrumbs would drop a fading marker at each event's actual played position when the audio plays, leaving a visible trail of recent events. This works for any pattern, deterministic or stochastic. It does not pre-show where the next firing will be, but for a tool where the composer is composing in real time and listening, the retrospective view is often more useful than the predictive one.

A hybrid is plausible: static glyphs at edit time for deterministic patterns, after-the-fact breadcrumbs for stochastic ones, with detection of which mode applies happening at parse time (a quick walk of the pattern's AST identifies stochastic modifiers). The final decision on which mode (or hybrid) ships is deferred until the rest of the visualisation surface gets focused attention during the Canvas consolidation pass.
