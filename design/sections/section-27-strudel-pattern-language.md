# Section 27 — Strudel as Pattern Language

Status: design intent for upcoming work. Captures the architectural decisions reached during the 2026-05-08 design session that established the new GXW project's direction. The user-facing surface — inspector layout, the cursor-as-collider model, visualisation choices — was originally part of this section and is now split out to section 28. Not yet implemented; subsequent commits will land the integration in tiers.

## Background and motivation

The earlier GXW codebase, now forked off as the GXSTR sibling project, integrated strudel as both pattern engine and audio engine, with strudel's bundled cyclist scheduler driving timing. Two extended attempts to reconcile that integration with GXW's free-running, physics-driven, cursor-and-trigger-fired event model surfaced a real architectural mismatch. Strudel's master cycle fundamentally wants to be the source of truth for everything plugged into it. GXW's clock is the simulation, with each curve running its own cycle at its own rate, sprites moving under physics, and triggers firing whenever a sprite happens into one. Trying to keep strudel's master cycle synchronised with GXW's per-object timing meant undoing the part of strudel that makes it useful as a scheduler; trying to let strudel drive meant collapsing GXW's free-running model into a master-cycle model that didn't fit its visual-and-physics-driven character. The two clocks were always going to chafe.

The shape described here is the alternative. GXW absorbs strudel's pattern-language layer and audio output without taking on its scheduler. GXW's simulation continues to be the master clock. Mini-notation strings, the Pattern abstraction, and modifier algebra all become first-class GXW concepts; superdough provides the audio rendering; the strudel cyclist is initialised for audio-context setup and sample-bank loading and then sits idle for the lifetime of the app.

## The architectural split

GXW owns the WHEN. Strudel owns the WHAT. That phrasing captures the layering precisely. The WHEN is the temporal structure: when a curve's cycle starts and ends, when its cursor crosses an active beat point, when a sprite collides with a trigger, when a free sprite's cycle wraps around. All of that is GXW's existing simulation, unchanged. The WHAT is the musical content of each firing: which sounds play, in what subdivisions, with what modulations. Mini-notation expresses that compactly, modifiers transform it, signals make it responsive to scene state.

The strudel library is really three separable layers stacked together: the mini-notation parser plus modifier algebra (the language layer), the Pattern abstraction itself (a pure data thing — a function from a time range to a list of Haps), and the cyclist plus superdough audio engine (the runtime layer). The library bundles them as a package, but they're cleanly separable. GXW absorbs the first two layers fully, replaces the cyclist with its own simulation-driven scheduling, and uses superdough as a black-box audio renderer driven by GXW's own scheduling rules. The Pattern type doesn't know or care that strudel's cyclist isn't running. queryArc is just a function — give it a cycle range, get back events. Whether a master clock is calling that function once per audio frame or GXW is calling it once when a curve's beat fires, the Pattern doesn't care.

## Pattern evaluation flow

When GXW's simulation produces a firing event for source X at simulation time T_cycle, the pattern-evaluation layer looks up the source's parsed Pattern (cached at edit-commit time and re-parsed when the user commits a pattern edit), calls pattern.queryArc(n, n+1) where n is the source's current per-source cycle counter, and receives a list of Haps. For each Hap, GXW computes a wall-clock event time T_event = T_cycle + e_t × cycle_duration, where e_t is the Hap's fractional position within [0, 1) and cycle_duration is the source's current wall-clock cycle period derived from master BPM and the source's cycleBeats. A callback is scheduled at T_event minus the audio scheduler's lookahead (Web Audio's recommended ~100ms forward window). The source's cycle counter increments, so n+1 becomes n on the next firing — this lets cross-cycle modifiers like alternation advance correctly.

Each callback, when it fires just before the audio's actual play time, sets a module-level firing-context pointer to (source X, T_event), re-queries the pattern with a tiny window around e_t for just this event, receives a Hap whose dynamic-signal values reflect simulation state at near-play time, calls superdough.play(T_event, hap.value) to render the audio, and clears the firing-context pointer.

The two-pass evaluation is the key technique. Pass 1, at cycle start, determines event TIMES from the pattern's structure. Pass 2, at near-play time, determines event VALUES with the firing context active. Static signals like sine return identical values in both passes since they depend only on cycle position. Dynamic signals like spriteV, spriteX, imageColor return different values in pass 2 because simulation state has advanced and the firing context lets them read current state. The per-event re-query cost is small — each is a tiny pattern evaluation against a sub-cycle window — and pays for accuracy: dynamic signals reflect simulation state at the event's actual play time, not at cycle-start time. For a wandering sprite firing image-color events at multiple positions in a cycle, each event reads the pixel under the sprite at the moment that event plays, give or take the audio lookahead window of about 100ms.

The unavoidable residual is the audio lookahead itself. Web Audio needs roughly 100ms of forward scheduling to avoid glitches, so dynamic signals are sampled 100ms before audio output rather than at the exact instant of audio output. For a wandering sprite that's a small fraction of a beat's worth of motion, far smaller than the per-cycle sampling delay would be. It's a Web Audio fact, not a strudel fact, and the two-pass architecture pushes it to its theoretical floor.

## Curated modifier vocabulary

GXW imports the full @strudel/web library and uses the parser, Pattern type, and modifier algebra as-is. The Pattern abstraction is general enough that modifiers compose normally; queryArc returns coherent results regardless of which modifiers a user composes into a pattern. What GXW promises and documents is a curated subset of modifiers known to behave intuitively under the per-source-cycle clock model. Other modifiers remain accessible for users who reach for them, and the inspector surfaces a soft warning when a modifier is used outside its promised context.

The within-cycle modifiers all work cleanly and are promoted as core vocabulary. Subdivision `[a b]`, repetition `a*4`, parallel `a,b`, weighted slots `a@2 b`, rest `~`, the `.fast(N)` modifier (which divides cycle time so more events fit), `.rev` (within-cycle reverse), `.struct(...)`, `.jux(fn)`, `.layer(...)`, and all the sound-selection and parameter setters — `.s`, `.n`, `.note`, `.gain`, `.pan`, `.lpf`, and friends. These produce events whose times and values are determined entirely within one cycle, so they need no continuous master-clock context to behave as the user expects.

Cross-cycle modifiers work fine when the firing source is continuous. A curve whose cursor loops forever has its cycle counter advance naturally on each wrap, so alternation `<a b c>` rotates through values across firings, `.every(4, fn)` applies a transformation every fourth firing, and `.iter(N)` rotates subdivisions across cycles. These are part of the promoted vocabulary for continuous sources.

Cross-cycle modifiers are problematic when the firing source is one-shot. A trigger collision or a sprite-hit-trigger event fires sporadically with no inherent next cycle. Under `.slow(N)`, only one Nth of the events fire per firing because the pattern is stretched across multiple cycles that the one-shot never sees. `.late(N)` and `.early(N)` shift events into adjacent cycles where the one-shot doesn't query, so events silently disappear. The inspector flags these patterns with a soft validation warning (yellow squiggle, matching the existing soft-error pattern for things like duplicate names) when the source's firing model is one-shot, but doesn't hard-reject them. A user who understands the implications can commit a warned pattern; the parse and evaluation are correct, just the typical intent doesn't match the result.

The decision is to use the full library and curate at the documentation and inspector-warning layer rather than fork strudel and ship a custom subset. The parser is substantial code, modifier definitions are intertwined with Pattern internals, and pulling out a curated subset cleanly is harder than it sounds. We'd be taking on a maintenance burden for no gain, and we'd lose the benefit of strudel as a known quantity that other people understand and have documentation for. Curation at the surface is the right shape.

## Static and dynamic signals

Strudel's built-in signals (sine, saw, square, tri, perlin) are pure mathematical functions of cycle position. They work identically under GXW's clock as under strudel's master cycle: queryArc samples them at each event's fractional position, and the value depends only on that position, not on wall-clock time. No modification needed; they integrate as-is.

Dynamic signals are GXW's contribution to the signal vocabulary. They aren't extensions to strudel; they're new Patterns whose query function happens to read external state during the call. Strudel's Pattern type is general enough to express them naturally — a Pattern is just a function from a time range to a list of Haps, and nothing about that function needs to be pure. Anything that implements the queryArc shape composes with mini-notation and modifiers identically to a built-in signal because it looks identical to the rest of the system.

The standard signal set falls into three groups. Image-colour signals derive from the OKLCh perceptually-uniform colour space and read the image pixel at the firing source's current canvas position: imageLightness (perceived brightness, calibrated so equal numerical changes correspond to equal perceived brightness changes), imageChroma (perceptual saturation, "how vivid versus grey"), imageRedness and imageGreenness (opposite ends of the red-green opponent axis, each the negation of the other so composers pick whichever sign matches their intent), imageYellowness and imageBlueness (the same shape on the yellow-blue opponent axis), and imageLum (a separate grayscale luminance convenience field in the Rec. 709 luma style, distinct from the perceptual imageLightness). Sprite kinematic signals: spriteX, spriteY, spriteVx, spriteVy, and spriteV (scalar speed). Harmony-context signals: currentScale, currentChord, currentTonic, currentRoot for the inherited harmony context (deferred until the tonal integration question is settled).

GXW exposes only OKLCh-derived perceptual colour as the standard colour-driven music vocabulary; raw RGB is not exposed in the standard signal set. The redness/greenness and yellowness/blueness pairs encode the perceptual opponent axes that human vision actually uses, and the named-positive-and-negative-of-each-axis form lets composers pick whichever sign reads naturally in context (imageRedness for "music responds to red regions," imageGreenness for the inverted intent). Hue is not exposed as an angle: the opponent-axis projections give composers smooth values that don't suffer wraparound discontinuity when a sprite traverses the colour boundary at 0 or 360 degrees, which an angular hue would. Composers who specifically want angular hue can compute it themselves with Math.atan2(yellowness, redness) inside a composer-defined signal (the defineSignal mechanism is deferred but tracked in TODO.md).

The mechanism for dynamic state lookup is the firing-context pointer described above. When GXW fires a source's pattern at simulation time T, it sets a module-level pointer to (source X, T) inside a try-finally block (so any exception during query doesn't strand the pointer), calls queryArc, and clears the pointer. Within the queryArc call, dynamic signals consult the pointer to learn which source to read state from, then read it from the live simulation. The Pattern composition machinery doesn't know or care that the value came from simulation state versus a sine function. Two-pass evaluation ensures dynamic signals are evaluated at near-play time, not at cycle-start time, so a wandering sprite's fast position changes within a cycle reach the audio output correctly.

## Tempo and scaling

The tempo model separates two concepts cleanly. Master BPM and per-source cycleBeats together determine the WALL-CLOCK PERIOD of each source's cycle. The mini-notation pattern determines the WITHIN-CYCLE EVENT DENSITY. Either can change without affecting the other.

At master 120 BPM, a curve with cycleBeats=4 has a cycle period of 2 seconds. With pattern `bd*4`, four events spread evenly across that 2-second window — events fire at the natural 120 BPM rate. With pattern `bd*8`, eight events in the same 2-second window — events fire at a 240 BPM rate within an unchanged cycle. With pattern `bd*2`, two events at 60 BPM rate. The user authors rhythm density and master tempo independently, the way a composer thinks about them.

A BPM change rescales every active cycle's wall-clock duration on the next cycle wrap, the same way the existing simulation already rescales cursor advance rates. A pattern edit changes what plays within the unchanged duration. No global re-scheduling, no glitches at the boundary, just per-cycle adaptation that flows through the existing simulation tick loop.

The mapping from pattern-cycle-relative position to wall-clock event time happens in GXW's runtime, not in strudel's modifier algebra. The runtime calls Pattern.queryArc to get events with fractional positions in the unit interval, multiplies each fractional position by the source's wall-clock cycle duration (cycleBeats times sixty divided by master BPM), and schedules at the resulting time. Strudel's slow, fast, and compress modifiers are not used in this path. The pattern stretches or compresses to fit each cycle's wall-clock window with no rests, no truncation, no time-stretching artefacts; the source effectively runs its own local clock at a rate forced by fitting the pattern into the wall-clock cycle. This direct-time-math approach also means GXW does not need to call strudel's setcps and is unaffected by the setcps reachability issue documented in the GXSTR project (where strudel was driving its own scheduler and required setcps to control rate).

## Continuous and one-shot firing

The same pattern-evaluation primitive serves both continuous and one-shot firing patterns, with different invocation behaviour at the call site.

Continuous firing applies to sources whose cycles repeat naturally. A curve whose cursor loops the curve, a free sprite with a cycle binding, anything where the simulation produces a regular sequence of cycle starts. For these, GXW calls the pattern primitive with sequential cycle counters (0, 1, 2, ...) on each cycle wrap, and the per-source cycle counter naturally advances cross-cycle modifiers like alternation.

One-shot firing applies to events that don't naturally repeat. A trigger collision, a sprite-hit-trigger event, anything where the user wants a flourish of music to play once at a moment in time. For these, GXW calls the primitive once with cycle counter zero and a user-chosen flourish duration mapping the pattern's [0, 1) cycle range onto a wall-clock window. The flourish duration is a per-source field defaulting to one beat at the master BPM, with override available per-source.

Both invocation patterns flow through the same queryArc-then-schedule-events machinery. The difference is just how often the primitive is called and what cycle counter is supplied. A trigger collision firing one-shot might still use rich within-cycle mini-notation (subdivisions, parallel layers, parameter modulation) — the pattern just doesn't get cross-cycle modifier behaviour.

## Open questions and future considerations

Tonal integration. Strudel ships with @strudel/tonal, providing harmony operators (scale, voicing, transpose) that sit naturally alongside mini-notation. Whether GXW imports tonal is open. The argument for is that harmony-aware patterns are a natural fit for an image-driven music model. The argument against is that the dependency surface grows and tonal hasn't surfaced as a need in concrete use cases yet. Defer until a real use case demands it.

Per-source flourish duration default. The default of "one beat at master BPM" for one-shot patterns is a starting guess. It might want to be a per-pattern setting (specified in the scene data rather than per-source), or different defaults per source kind (triggers versus sprite-hit events versus auto-fire intervals). Real use will tell us which shape feels right.

Concurrent firings. Two sources firing patterns at the same simulation tick is fine in principle — each callback runs synchronously with its own firing-context-set / queryArc / clear / superdough.play sequence — but per-source cycle counter management when many sources fire concurrently has implications for cross-cycle modifier behaviour that haven't been thought through end-to-end. Likely fine in practice; worth confirming when the implementation lands.

Mute and solo. GXSTR has Solo Selected as a runtime-only state and per-source mute as a persisted state. Both should carry over to GXW's pattern integration; the pattern-evaluation layer respects mute and solo by skipping queryArc entirely when the source is silenced. Implementation is straightforward but worth calling out.

Edit-time pattern preview. The user types a pattern in the inspector field and probably wants to hear it before the next firing in the simulation. A "preview pattern" affordance — play this pattern once with a small flourish window, right now — is a natural extension. Implementation is a one-shot invocation of the same primitive, triggered by a button or keyboard shortcut on the field. Inherits the user-gesture-for-AudioContext requirement, so a first preview also serves as the engine-load gesture.

Recording and replay. Resolved: GXW commits to determinism as a guiding principle. The simulation evolves as a function of initial scene state plus elapsed tick count, with no dependence on wall-clock timing during a run. AudioContext.currentTime drives the tick clock, but the simulation reads tick count and tick-derived position rather than wall-clock time. Rewinding to time zero replays sample-identical audio on every replay. Stochastic mini-notation modifiers (degradeBy, scramble, choose, the rand and irand signals) are seeded by a per-score random number generator. The seed is part of the score's persisted state. New scores get a fixed default seed; the seed-changing UI is hidden from default users and surfaces later as an expert affordance in a score-settings dialog. Tier 1's StrudelRuntime initialises a mulberry32 PRNG with the score's seed and exposes it on window.gxwRandom; whether strudel itself reads from this RNG or whether we have to monkey-patch its internal random source is a discovery question that lands in Tier 2 when stochastic modifiers first see actual use. (Resolved.)

Sample bank loading. GXSTR loads dirt-samples (drums) and dough-samples piano.json at engine init time. The same loading carries over to GXW's pattern integration, with the same Load Engine button pattern from GXSTR — the user clicks to satisfy the browser's audio-context gesture requirement and lazy-load samples — applying here.

Mini-notation autocomplete in the inspector. The inspector's pattern field could surface mini-notation completions (sample names from loaded banks, common modifier names, signal names) similar to the editor's CodeMirror completions in the Behaviours tab. Out of scope for the initial integration; a natural follow-up once the basic surface is working.

Strudel version sync. GXW pins to a specific @strudel/web version (matching whatever GXSTR currently uses, since the import path is shared). Strudel evolves; staying current on its releases is an ongoing maintenance concern, not a one-time decision. Worth tracking changes that affect the modifier vocabulary GXW promises.

## Pre-implementation design tasks

Six tightening points need to be resolved before Tier 1 begins. They surfaced during a design sanity-check pass after the section as originally written, and they mark places where the design is currently too hand-wavy for clean implementation. They take precedence over other open questions and over any review of sections 1 through 26, because some pre-pivot work items may become irrelevant or need rewriting once these are settled.

The first is empirical verification of the layer separation. The cleanness of strudel's three-layer structure (mini-notation parser plus modifier algebra, Pattern abstraction, cyclist plus superdough audio) is asserted earlier in this section but has not been verified against running code. A small spike, before any real Tier 1 work, that imports @strudel/web, parses a pattern, calls queryArc directly without ever starting the cyclist, and pipes the resulting Haps to superdough manually, would prove or disprove the separation in an afternoon. The most likely places hidden coupling could surface are global state the cyclist normally maintains (window.cps and similar) and superdough's expectations about the cadence at which it gets called. If something unexpected does turn out to be cyclist-coupled, we want to find that on day one rather than at Tier 4. The spike's output is either a green light to proceed with the rest of Tier 1 as designed, or a list of concrete coupling points that need to be addressed before Tier 1 is realistic.

The second is the Pass 2 edge cases. Pass 1 of the two-pass evaluation establishes event times from the pattern's structure. Pass 2 re-queries a tiny window at near-play time to read dynamic signal values. The section as originally written assumes Pass 2 returns the same event Pass 1 found with potentially refreshed value fields, but does not specify what happens if Pass 2 returns no event (because some modifier's behaviour depends on something that has changed in the meantime) or returns multiple events. Decision: trust Pass 1's structure absolutely and use Pass 2 only to refresh value fields. If Pass 2 returns no event in the window or returns multiple events, the schedule keeps Pass 1's event with Pass 1's values. This rule keeps the schedule deterministic relative to cycle-start state and isolates dynamic signal influence to value fields, which is the desired behaviour. (Resolved.)

The third is cancellation of pre-scheduled events when a pattern changes mid-cycle. If a curve is firing pattern A and the user commits an edit to pattern B while events from A are already in superdough's forward window, those events will fire unless cancelled. Web Audio supports cancellation via cancelScheduledValues on parameters and via stop on AudioBufferSourceNodes. Superdough may or may not expose these cleanly. Decision (provisionally): always cancel events in flight at the edit point so the new pattern takes effect immediately, with revisit deferred until a working integration could be heard. Revisited and resolved differently once Phase 2 of the audio firing path landed: the firing engine does NOT cancel events on a pattern change. A mid-cycle takeover (clearing pending events and re-detecting against the new pattern within the same cycle) was tried and reverted because the blended audio (old beats from before the edit plus new beats from after) sounded messier than waiting one cycle for a clean boundary transition. A take-effect-on-next-wrap shape that cleared pending events at the edit point was also tried and rejected because it produced a silence gap from the edit through to the next wrap. The shipped resolution preserves pending events: the old pattern's already-queued events for the rest of the current cycle play through to the cycle boundary, and the new pattern takes effect at the next wrap via the normal detect-and-populate path. No silence gap and no audio messiness. Superdough cancellation primitives remain unused; the Tier 5 one-shot work will need cancellation machinery for trigger-deletion edge cases but it is now decoupled from this Tier 2 pattern-edit case. (Resolved.)

The fourth is per-source cycle counter persistence across edits. If a curve's cycle counter is at 47 when the user edits the pattern, does the counter reset to 0 on the next wrap, or continue at 48? For cross-cycle modifiers like alternation, this is the difference between an edit causing alternation to restart at the first slot and an edit just changing what plays from now on, with alternation continuing from where it was. Decision: continue-on-edit. The counter is not reset by a pattern edit; alternation and other cross-cycle modifiers progress through whatever the new pattern does on the next cycle index. An explicit reset gesture (a small inspector control that resets a source's counter to 0) covers the rare case where the user wants to restart alternation deliberately. Confirmed in Phase 2 of the audio firing path implementation: the firing engine preserves lastFiredCycle across pattern recompile, and an alternation pattern edited mid-cycle picks up at the next cycle index of the new pattern, not at cycle 0. (Resolved.)

The fifth is the relationship between wall-clock time and simulation time. GXW's simulation has its own tick clock that may run faster, slower, or at variable rate compared to performance.now or audioContext.currentTime. The scheduling math in the Pattern evaluation flow section (T_event = T_cycle + e_t times cycle_duration) assumes simulation time and audio time are in lockstep, but they are not in general. A simulation pause, a slow frame, or a user-driven scrub will desynchronise them. The design needs to specify which clock drives the schedule. Audio time is the right answer for events that need to land on time without glitches; simulation time is the right answer for visual coherence.

Decision: a hybrid driven by the simulation tick. At cycle start (a per-source moment determined by the simulation, since each source's cycle can be anywhere from a fraction of a second to many seconds depending on master BPM, the source's cycleBeats, and any per-source tempo scaling), the pattern is queried once via queryArc to produce the event list for that cycle and the cycle's audio start time is captured. Events are NOT pre-scheduled to Web Audio for the whole cycle. Instead each simulation tick walks the active sources, finds any events whose audio play time falls within a small forward window (the audio commit window, on the order of 100 milliseconds, sized to be larger than one tick interval but small enough to keep simulation-pause behaviour responsive), and commits just those events to Web Audio via superdough's absolute scheduling. Anything beyond the window stays on the per-source pending list until a later tick reaches it.

This hybrid handles every clock-skew case cleanly. A short cycle has its full event list committed in one or two ticks. A long cycle has events committed in rolling forward windows over the full cycle duration; nothing is queued in the audio graph more than the commit window ahead. A simulation pause stops the ticks, no new events commit to Web Audio, and audio tails off after the last committed window finishes; resume picks up where we left off. A scrub is treated as a non-real-time interaction and audio is allowed to silence during the scrub. The cycle-start alignment to simulation time preserves visual coherence at the boundaries between cycles, while the audio-time scheduling within the commit window keeps within-cycle events crisp against the audio clock.

The specific size of the audio commit window is left to be tuned in practice. 100 milliseconds is the natural starting value because it matches Web Audio's recommended forward scheduling window and what strudel's own cyclist uses, but the right number depends on simulation tick rate and how responsive the pause behaviour feels under load. The StrudelRuntime stub should expose the window size as a configurable value so it can be adjusted without code changes once the integration is running. (Resolved.)

The sixth is strudel's global-state setup. Even though we never run the cyclist, some modifiers in the promoted vocabulary may consult globals like window.cps that the cyclist normally maintains. Decision: proactive. Tier 1's StrudelRuntime stub includes a defensive global-state initialisation step that sets the well-known globals (window.cps derived from master BPM is the most important; others added as discovery surfaces them) and exposes them as configurable values driven from master BPM and per-source cycleBeats rather than from the unused cyclist. The init step is a small block of code at runtime startup that costs nothing and protects against the most likely class of silent breakage.

Related: some modifiers may behave incorrectly under the per-source clock model regardless of how globals are set, because their semantics assume strudel's master cycle. Once Tier 2 and Tier 3 are running we may identify specific modifiers that should be disabled outright or marked as unsupported in the inspector's modifier vocabulary documentation, separate from the soft-warning machinery for context-inappropriate use that section 27 already specifies. The decision on which (if any) need this treatment is deferred until the integration is running and the actual behaviours can be observed. (Resolved.)

All six items above have now been resolved. Items 1, 2, 5, and 6 are firm decisions from the pre-implementation pass; items 3 and 4 were provisional at the time and have since been resolved firmly through Phase 2 of the audio firing path implementation (item 3 to no-cancellation-preserve-pending, item 4 to continue-on-edit confirmed in practice). Tier 1 and Tier 2 Phases 1 and 2 have all landed from this point.

## Audio firing path: full strudel, phased rollout

A choice between two paths surfaced during pre-implementation planning, after Tier 1 landed and before Tier 2 work began. The first was the path this section describes: GXW builds the full pattern evaluation primitive with firing-context pointer plumbing, two-pass evaluation, and per-tick state snapshot mechanism, every piece of the runtime architecture above, landing eventually in working dynamic signals through strudel's vocabulary. The second was a simpler intermediate: GXW does not fire strudel patterns at all and instead calls user-named JavaScript functions on cycle wrap, letting the user author audio via Web Audio or Web MIDI directly. The simpler path lets sound come out of the system faster but at the cost of throwing the strudel-vocabulary benefit away on the audio side; most of it becomes throw-away code when the full path lands later. A hybrid was also considered (implement the deterministic-and-static-signal portion of the strudel path now, defer the dynamic-signal portion as a separate later chunk) but it just describes the natural Phase 2-to-Phase 3 boundary inside the full path rather than a different path.

Decision: the full strudel path. The project's musical intent — patterns whose audio output depends on scene state changing during firing, the strudel modifier vocabulary applied to authored mini-notation, harmonic context via the eventual @strudel/tonal integration — needs the strudel pattern primitive working end-to-end. Anything less leaves GXW conceptually short of its goal: a music app whose patterns interact with the canvas through dynamic signals, not just a sequencer that renders mini-notation in lockstep with cursor sweeps. (Resolved.)

Implementation in four phases so each step is testable before the next builds on it. Phase 1 lands the minimum to fire a deterministic pattern through superdough on a curve cursor sweep: pattern primitive function, per-source cycle counter, audio commit window, mute and solo gating, continuous firing path. Static signals (sine, saw, square, tri, perlin) work for free at this stage because they are pure functions of cycle position. Phase 2 lands the edit-time polish: a clean cycle-boundary takeover when the user edits a pattern (old pattern's pending events play through the rest of the current cycle, new pattern takes effect at the next wrap, no silence gap and no blended-audio messiness), reliable position-zero firing on every cycle wrap (one-cycle-ahead scheduling: each tick the engine populates events for both the current cycle and the next cycle, so the position-zero event of cycle N+1 is queued well before the simulation actually wraps to N+1; Web MIDI schedules it accurately at its true audio time with no detection-lag stutter at cycle boundaries), and editor undo/redo across Cmd-Enter promote (the editor's refresh-from-bundle hook skips its CodeMirror dispatch when the active tab's content is already in sync, preserving the redo stack). MIDI output integration also lands in Phase 2 as the default output path: the firing engine routes events through a MIDISender adapter to a Web MIDI port (an IAC bus on macOS) so users can drive external software synthesisers rather than being constrained to strudel's built-in samples. Phase 3 lands the dynamic-signal substrate without any new signals: firing-context pointer in try-finally, two-pass evaluation reconciled with one-cycle-ahead scheduling via a late-refresh dispatch (described below), per-tick state snapshot. Nothing observable changes at Phase 3; the infrastructure regression surface is intentionally decoupled from the new-signal behaviour so a regression in either is debuggable without confusion from the other. Phase 4 lands the first dynamic signals. The previous ordering put sprite kinematics first on the convenience argument that the simulation already carries velocity and position data; revisited, the right musical argument is to put the image-colour signals first because sprite velocity has no varying state to read until physics is driven by pixel data (a sprite under the current simulation moves at its authored velocity between wall bounces and produces a flat signal). The new ordering: imageLightness first as a clean scalar that exercises the full pixel-lookup and OKLCh-conversion pipeline; the other image-colour signals (imageChroma, imageLum, and the four opponent-axis signals imageRedness, imageGreenness, imageYellowness, imageBlueness) follow in close succession because they share the conversion pipeline and each is a trivial wrapper once OKLCh is in place. Sprite kinematic signals (spriteX, spriteY, spriteV) come as a separate later step once the use case for them comes back into view, likely when image-driven physics lands and velocity becomes a meaningfully varying value. Distance-derivatives of the image signals (dlightness_ds, dredness_ds, and so on) and EMA smoothing of values are deferred to a follow-up phase; both are layered on top of the basic plumbing rather than altering it. Phase 4 overlaps with what the Implementation order subsection below captured as Tier 4 in the original taxonomy; the overlap is intentional now that the plumbing dynamic signals consume lives in Tier 2 rather than being deferred.

The natural stop-and-back-out point is between Phase 2 and Phase 3. If Phase 1 and 2 land cleanly but dynamic-signal integration turns out intractable, the system is left in a working hybrid state — full deterministic and static-signal patterns work, dynamic signals do not. Not abandoning the strudel path, just stopping at the deterministic point while the issue is worked out. This is the fallback that makes choosing the full path safer than it would otherwise be: a problem at Phase 3 or 4 doesn't unwind back to Tier 1.

## One-cycle-ahead scheduling and dynamic-signal late-refresh

The Pattern evaluation flow subsection above describes the original two-pass design: events are scheduled at cycle start with audioTimes derived from the pattern's structure, and a tiny per-event re-query at near-play time refreshes value fields with the firing context active. Implementation experience in Phase 2 of the audio firing path landed a related but slightly different architecture worth documenting here, because the two interact non-trivially.

One-cycle-ahead scheduling. The original design's Pass 1 scheduled events for the current cycle at cycle start. In practice, detection of a cycle wrap necessarily happens slightly after the wrap itself: the simulation's 240Hz fixed-step tick overshoots the cycle boundary by 1 to 2 steps (about 4 to 8 ms), then the render loop reads it 0 to 16 ms later (one frame at 60fps). By the time the firing engine sees the wrap, the position-zero event's intended audioTime is about 20 to 40 ms in the past. Scheduling it through Web MIDI at that point either fires it late (the audio output rejects past timestamps and emits the note immediately, audibly delayed) or requires a forward-clamp that ALSO produces an audible offset. Either way, the cycle's downbeat lands consistently late and produces an audible stutter at every cycle boundary.

The fix is to populate cycle N+1's events during cycle N. On the tick that detects a wrap into cycle C, the firing engine ensures both cycle C and cycle C+1 are in its populatedCycles map. The cycle C+1 entry was created on the previous cycle's pre-population pass, so its position-zero event has already been queued at its true future audioTime. When the simulation actually wraps to C+1 (about 25 ms after the true cycle boundary), the event has already been sent to Web MIDI well in advance of its scheduled time; Web MIDI dispatches it exactly at the cycle boundary as intended. No detection lag, no clamp needed, no audible stutter.

The pattern-edit case (a cyclePattern text change committed via Cmd-Enter) interacts with the pre-populated next cycle. If the user edits the pattern at fractional position 0.4 of cycle C, the events for cycle C+1 were populated from the OLD pattern at cycle C's start. Letting those pre-populated events fire would mean one full cycle of old pattern audio after the edit, which is not Version B's clean-takeover behaviour. So _reconcileSource flags the source as patternDirty and the next tick filters pendingEvents to drop any with cycleIndex strictly greater than the current cycle, drops future cycles from populatedCycles, and clears the flag. The current cycle's events stay queued for the rest of the cycle (Version B); cycle C+1 is then re-populated from the new pattern on this same tick, giving the new pattern at the upcoming cycle boundary as intended.

Dynamic-signal late refresh. The one-cycle-ahead scheduling creates a tension with dynamic signals like spriteLightness, spriteV, imageColor. These signals read simulation state at evaluation time. A note whose audioTime is 1 to 2 seconds in the future was scheduled with the dynamic signal evaluated at population time (when the sprite was at its previous position), but should fire with the value reflecting the sprite's position AT the event's actual time. Resolving this requires the two-pass evaluation from the original design, reshaped for the pre-population architecture: Pass 1 happens at population time and establishes event structure (note number, audioTime, duration, channel, and any static-signal value fields that depend only on cycle position). Pass 2 happens just before each event's audioTime and refreshes value fields that read dynamic signals.

The specific dispatch mechanism for Pass 2 with MIDI output: instead of pre-sending the noteOn message to Web MIDI at population time, the firing engine retains the populated event in pendingEvents until a tight window (say 20 ms) before its audioTime. At that window, the commit walker activates the firing context for this event, re-queries the pattern for a tiny range around the event's fractional position, takes the refreshed value, and dispatches the noteOn at that moment with Web MIDI's send-now semantics (using the converted timestamp for audioTime, which is by now within milliseconds of the present). Web MIDI's own scheduling latency is ~1 ms in practice, so the event lands at audioTime within audible threshold.

This hybrid keeps the timing benefit of one-cycle-ahead populate (cycle-start audioTimes are computed in advance, no detection-lag stutter at wrap boundaries) AND the freshness benefit of late-refresh dispatch (dynamic signal values reflect simulation state at audible-near-play time). The commit walker becomes the late-refresh dispatcher rather than just an audioTime-window filter; its inputs are pendingEvents and audioNow, its outputs are MIDI sends with Pass-2-refreshed values. For events with no dynamic-signal-dependent value fields, Pass 2 is a no-op and the dispatch is functionally identical to the current Phase 2 commit path. The performance cost is one queryArc per event in flight, restricted to the moments just before each audio dispatch — the same cost as the original design's Pass 2, just unified with the pre-population queueing.

The noteOff dispatch is more subtle. With the duration multiplied by value.clip, a noteOff for an event with a long sustain (legato 2.0, for instance) could be 4 seconds after the noteOn. Should noteOff also Pass 2 refresh? Conservative answer: no. The dynamic-signal value at noteOff time would only matter for controlling sustain behaviour, which MIDI noteOff messages don't typically express (velocity-on-noteOff is rare and synth-specific). The noteOff dispatches at noteOn time plus duration * clip with no refresh, using the Pass 2 value already captured for the noteOn pair. If a synth-specific use case for noteOff late-refresh emerges, the mechanism is straightforward to extend.

CC streams (continuous controllers) for dynamic-signal-driven modulation are where Pass 2 refresh matters most. A pattern like note("c4").ccv(spriteV.range(0, 127)).ccn(1) sends a stream of mod wheel values driven by sprite velocity. Each CC event in the stream needs Pass 2 refresh so the value reflects current sprite velocity, not the velocity captured at population time. The Pass 2 mechanism applies uniformly: each CC event in pendingEvents waits in the queue, gets Pass-2-refreshed just before its audioTime, and dispatches at the refreshed value. The result is a smooth modulation stream that tracks live simulation state at the audible threshold.

This subsection supplements rather than replaces the Pattern evaluation flow description above. The earlier description's two-pass framing is still correct; this subsection just maps it concretely onto the one-cycle-ahead populate architecture that Phase 2 landed, and onto the Web MIDI output path that Phase 2 also landed.

## Implementation order

When implementation begins, a tiered approach mirroring GXSTR's phase plan is the natural shape.

Tier 1 lands the foundations: import map for @strudel/web and tonal, LICENSE update, basic StrudelRuntime stub that can initialise the audio context and load sample banks but doesn't yet drive any output.

Tier 2 lands the pattern evaluation primitive: a function that takes a parsed Pattern, a cycle range, a wall-clock window, and a firing source identifier, and produces scheduled superdough events with the firing-context pointer mechanics in place. Static signals work end-to-end; dynamic signals not yet wired.

Tier 3 lands the inspector integration: the fourth beat-points-mode option, the pattern field, parse validation, soft-warning validation for context-inappropriate modifiers. Continuous firing on curves works end-to-end with static signals.

Tier 4 lands dynamic signals: SignalRegistry, the firing-context pointer plumbing, and the standard signal set with imageLightness as the first end-to-end signal landed (the firing-context plumbing landed in Tier 2 Phase 3 already; Tier 4 / Phase 4 builds on that). See the Audio firing path subsection above for the per-signal phasing within Phase 4.

Tier 5 lands one-shot firing: trigger collisions and sprite-hit events fire patterns through the same primitive, with the flourish-duration field added to the relevant inspector surfaces.

Tier 6 lands visualisation: pre-firing glyphs for deterministic patterns, after-the-fact breadcrumbs for the rest. Polish, edge cases, and the open questions above get addressed as they surface.
