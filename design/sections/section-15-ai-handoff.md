## Section 15 — Disk Mirror and AI Integration

The disk mirror is the mechanism by which an AI assistant — typically Claude Desktop with a filesystem MCP server pointed at the mirror folder — observes and edits the score the user is currently working on. The mirror is a single folder at a stable, known location holding a live copy of the open score's content: scene.json, behaviours.js, the image, optional snapshot images of the canvas, and a small metadata file naming the score. The mirror is kept in sync with the in-memory bundle as the user edits, on a debounced timer rather than on every keystroke. The AI reads from the mirror to understand the user's current state and writes to the mirror to propose edits, which flow back into the editor as bundle changes. The mirror is always running; there is no Hand Off or Fetch gesture to invoke, and the user does not interact with the mirror folder directly.

Status and history. This design reverses the v2.2 deprecation of an earlier disk-mirror approach and supersedes the v2.3 AI Handoff feature (explicit user-driven Hand Off and Fetch via a configured folder) that replaced it. The earlier disk-mirror was deprecated because it ran in a browser context: the File System Access API revoked filesystem permissions on every tab restart, prompting the user repeatedly; absolute paths were not exposed to the AI; module caches masked external edits until a hard reload; and the round-trip introduced subtle formatting bugs. None of those constraints apply in the Electron build, where the renderer talks directly to the filesystem through the IPC layer and scores live as real .gxs folders at user-chosen paths. The disk mirror is therefore Electron-only by design and not implemented in the web build; web users do not get AI integration through this path. The composition mirror described in this section is implemented in stage 4. A planned future phase adds runtime logging during playback (Section 15.1 below).

Location. The mirror folder is at ~/Library/Application Support/GeoSonel/Active/. The user configures their MCP server (or other AI tooling) to read and write this path once, and otherwise never touches it. ~/Library is hidden by default in Finder, so the mirror is not surfaced to the user during ordinary use; the path exists for AI access only. This keeps the user from accidentally editing mirror contents directly, which would not be the right way to modify a score (the in-memory bundle is the source of truth; mirror edits feed back into it but are not how the user should think about authoring). The Active folder always contains exactly the currently-open score; on a score switch the folder is cleared and refilled with the new score's content. The mirror does not persist history between sessions; each launch starts fresh with whatever the resolved initial bundle's state is.

Contents. The Active folder holds:

- scene.json — the score's composition data, current in-memory state.
- behaviours.js — the score's per-object callback code.
- image.png (or the bundle's actual image filename) — the score's background image.
- snapshot.png — a capture of the canvas's current pixels at last sync, showing where cursors, sprites, triggers, and selection markers are right now. Observation-only; the AI does not write back here.
- snapshot-annotated.png — the same capture with object indices labelled next to each visible element, so the AI can refer to specific objects by the same indices scene.json uses. Observation-only.
- active-score.json — a metadata file naming the score's display name, its real on-disk path (or null for Untitled), its dirty state, the timestamp of the last sync to the mirror, transport state (current play state, elapsed seconds, beat number), and a list flagging which files round-trip on AI edits versus which are observation-only.

active-score.json gives the AI the information it needs to know what it is looking at and where the user's authoritative copy lives. For an Untitled bundle, path is null and the AI knows the score has no canonical home yet; for a saved bundle, the AI can point its conversational reference at the user's filename ("In your Foo.gxs score, I see that ball A...").

Update model. Bundle state pushes to the mirror folder on a debounced timer — roughly 500ms after the last user edit — so the mirror reflects the user's current work rather than the last saved version. Switching scores, opening a different score, or creating a new Untitled bundle replaces the folder contents immediately rather than waiting on the debounce. The user's explicit Cmd-S still writes to the score's real on-disk path; the mirror tracks the in-memory state regardless of whether the user has saved. This means the AI can see and reason about work-in-progress, not just committed work, which matters most for Untitled bundles that have no on-disk path yet.

Bidirectional flow. An fs.watch on the Active folder picks up external writes (AI edits) and pushes them into the in-memory bundle through the same parse-mutate-stringify pipeline the canvas toolbar, JSON tab, and form-based property inspector all use (see Section 14, Section 13, Section 25 question 3). The bundle's dirty flag flips on; the user can then Cmd-S to commit the AI's changes to the score's real path. The pipeline does not echo the bundle's own writes back into itself: the watcher ignores changes whose mtime falls within the just-completed push window, so a write from the bundle to the mirror does not trigger a downstream pull. The active-score.json file is rewritten on every push and ignored on every pull; the AI cannot modify the bundle's identity through the mirror.

Conflict handling. When the AI writes scene.json or behaviours.js while the user is also editing, the AI's writes take precedence. The bundle's in-memory state is replaced, and the user is notified through the message area ("Claude edited scene.json — your unsaved changes have been replaced"). The user can undo the loss through the normal undo history; the AI cannot undo. This asymmetry reflects the broader pattern that the AI is the slower, more deliberate collaborator and the user's session has rich undo while the AI's session does not. Race-condition windows are kept narrow by the debounce on the upstream side and by single-threaded application of incoming writes on the downstream side.

Playback relationship. The composition mirror updates on bundle edits, not on simulation ticks. During playback, scene.json and behaviours.js in the mirror remain at whatever the user last edited them to be; the moving simulation state (object positions, velocities, current beat) does not flow into the mirror. This keeps scene.json's role as a schema unambiguous — the AI reading scene.json knows it is reading the composition, not a snapshot of a running performance. snapshot.png does update during playback (it captures the canvas pixels including any moving cursors and sprites), so the AI can still see what is on screen at any given moment, but the source files remain stable. If the AI writes scene.json during playback, the editor pauses playback, applies the change, and notifies the user; this prevents object positions from jumping unexpectedly mid-run as a result of an AI edit.

Workflow shape. The mirror is most ergonomic with Claude Desktop's filesystem MCP server pointing at the Active folder. The user works in GeoSonel as normal; Claude reads the mirror to understand the current score and writes to it to propose edits. The user sees Claude's changes appear in the editor within the same debounce window, can accept them by saving or revert them through the undo history. A bidirectional conversation about the score does not require any explicit Hand Off / Fetch gestures, which was a friction point in the v2.3 AI Handoff design. Other AI tools (cursor's MCP, ChatGPT with filesystem access, custom scripts) can use the same folder; the mirror is tooling-agnostic.

### Section 15.1 — Runtime Log (planned future phase)

During playback, the simulation can write a runtime log capturing object trajectories and note events. The log is a separate artefact from the composition mirror — describing what happened during a run, not what is being authored — and lets the AI answer questions like "where did ball A end up at the end of the piece?", "which objects play notes above C6?", and "why did ball B collide with field 3 when I didn't expect it to?". The runtime log is a planned future phase, scheduled after the composition mirror is stable and in real use.

Two streams interleaved by time make up the log. Frames sample the simulation state at a fixed interval (around ten samples per second is a reasonable starting point) and contain the time value plus per-object positions and velocities. Events are discrete: collision events (which two objects, at what time, at what position, with what incoming velocity), note events (which object, MIDI number and note name, velocity, trigger reason — collision with which field, beat tick, message function call), and any other discrete simulation outputs the AI might query. The trigger-reason field in note events is what turns the log from a transcript into a diagnostic tool: the AI can answer "why did this note play?" by reading a single field rather than reconstructing causality from positions. Sustained notes follow MIDI's note-on / note-off model as two separate event lines tied by an event ID, so the AI can compute durations directly and the firing system does not have to commit to a duration at note-on time.

Combined-query power is the main reason for logging trajectories and notes together rather than separately. Almost every interesting compositional question is a join across both streams: which object's path produces the bass line, which collisions trigger the lead voice, why the texture thickens at the second minute. The AI joining motion data and note data lets it talk about the composition as a coupled motion-music system rather than as two parallel things it cannot relate. Either stream alone would be substantially less useful than the pair, so the log lands them together rather than as separate phases.

Storage shape and filterability. The log must be queryable without the AI loading it all into context. A few minutes of playback at ten frames per second across twenty objects produces megabytes of NDJSON, which would consume significant conversation context if read wholesale. The storage choice settles when the runtime log is implemented; three candidate approaches are worth holding open. The first is SQLite as the primary store with the AI running queries through a small helper CLI tool — most powerful, requires the helper tool to exist. The second is pre-computed slices written alongside the raw log (notes-by-object/{id}.ndjson, notes-by-pitch-class/{name}.ndjson, frames sliced by time range), so the AI loads only the slice relevant to its question without needing a query tool. The third is a hybrid where a small summary.json is always present for fast preflight (run length, frame count, note count, pitch range, object IDs, duration) and the deeper data is stored in whichever format proves most ergonomic in practice. summary.json is cheap to compute and tiny to read regardless of which deeper-storage approach wins, so it likely belongs in the design either way.

Lifecycle. Each playback run produces a fresh runtime log, replacing the previous one. At any moment the user has the composition files plus a record of the most recent run. If history turns out to be useful — comparing the trajectory from one run to the one before the user tweaked a force field — keeping the last N runs as scene-runtime-{timestamp}.ndjson alongside a scene-runtime.ndjson pointer at the most recent is a natural refinement. The initial design ships latest-only.

Implementation note. The runtime log has to be written asynchronously off the simulation thread, or at minimum buffered so it does not introduce jitter in the playback. The buffer can be flushed every few frames or every wall-clock tenth of a second; the user should not be able to feel the recording happening. This is well-trodden ground — the same problem any DAW solves for its automation recording — and does not need to be tackled in detail until the runtime log is being built.

Exploration direction: history as a compositional substrate. The runtime log records what has happened in a run, primarily for AI inspection. A natural extension is to expose the same recorded history to the scene itself, so behaviour callbacks can query the recent past and use it as material for further composition. An object could echo notes it played four beats ago, transpose them up a fifth, and re-fire them; a second object could follow the spatial trajectory of a first object with a half-second delay; a third could play retrograde — its own past events in reverse — over a fixed window. Looper-style and music-transformer-style patterns fall out naturally once the history is queryable from within the simulation. Canonic imitation, inversion, retrograde, path mimicry, echo with decay, and density-driven response are all expressible as queries against the recent past combined with the existing callback model.

The architectural shift this implies is that history becomes a first-class feature in the simulation, and the disk log becomes one projection of it (the asynchronous persistent one, for AI access) rather than the primary representation. The simulation maintains an in-memory history buffer with a configurable retention window (typical values from a few beats to a couple of minutes; the composer picks based on what the scene needs); behaviour callbacks query this buffer synchronously through a narrow API like ctx.history.notesOf(object).inLast(beats: 4) returning past note events, ctx.history.positionOf(object).at(time: now - 2.0) returning past spatial positions, and ctx.history.collisionsOf(object).since(time: 0) returning the full collision record. The disk writer flushes the same data asynchronously to scene-runtime.ndjson without coupling to the synchronous query path. Query results are read-only arrays; the simulation does not let callbacks mutate the historical record.

This is captured here as an exploration direction rather than a committed feature. The composition mirror (Section 15 above) is the immediate stage 4 work. The runtime log without the in-simulation query surface is the natural next phase after that, primarily for AI access. The history-as-substrate extension above is a second layer on top, worth detailed design when both the composition mirror and the basic runtime log are in real use and the shape of useful compositional queries is clearer from practice. The current note exists so the architectural choice between log-as-disk-artifact and log-as-in-memory-feature is made deliberately when the runtime log lands, rather than locking in the disk-artifact framing before the compositional use is considered.

Exploration direction: phrase extraction. The user listening to a generated score often hears something they want to keep — a bass line between 8 and 10 seconds, a melodic phrase a particular object happened to play, the rhythmic interlock at a moment of high density. The runtime log records all the material needed to recover that phrase exactly: which object played, what notes at what timestamps with what velocities, what was firing in parallel. A phrase-extraction feature would let the user (or the AI on the user's behalf) query a time window with an optional object filter and write the result out as a portable musical artifact for use in other compositions — inside GeoSonel, in another DAW, or as a sketch in any other tool. This pattern has a lineage in KeyKit's snarf operation: the basic idea that what just played should be a first-class object the composer can grab and redeploy, with the runtime log giving GeoSonel a richer underlying record than older systems could rely on.

The interesting design surface is the output format, since the answer depends on where the phrase is going next. A standard MIDI file is the universal bridge — opens in any DAW or notation program, carries pitch and timing directly, so a captured fragment can be the seed of further work in whatever tool the user already uses. A Strudel mini-notation pattern serves the self-contained GeoSonel workflow, letting a phrase from one score paste verbatim into another score's pattern code. A GeoSonel-native clips format (a small file in the bundle's resources folder) could preserve fuller metadata — which object played what, trigger reasons, harmonic context — for cases where the phrase needs to round-trip back into GeoSonel with full fidelity. The right initial set is probably MIDI plus a Strudel pattern; the bundle-native format is worth adding only if the round-trip-to-GeoSonel use case proves common in practice.

Invocation has two natural paths. The direct user gesture is a region-select on the transport timeline followed by an Extract Phrase command that writes the selection to a clips folder. The AI-mediated path is conversational — the user describes the moment to Claude in natural language ("the bass line right after the first big collision"), Claude reads the runtime log to identify the relevant events, and writes the extracted phrase out as a file. The AI path is the more powerful one because the user can describe by feel rather than by precise timestamps; the direct gesture path covers cases where the user already knows exactly what region they want.

Phrase extraction and the history-as-substrate extension share a data model. Both are queries against the recorded runtime; one produces a callback-readable in-memory array for the simulation to consume during the same run, the other produces an on-disk file for use outside this run. Designing the runtime log so it can serve both consumers — a synchronous, narrowly-scoped query API for the simulation and an asynchronous, file-producing export API for phrase capture — is the architectural decision the runtime-log stage commits to. Both extensions then follow as natural elaborations of the same substrate.

Like the history-as-substrate extension above, phrase extraction is captured here as an exploration direction rather than a committed feature. Detailed design happens when the basic runtime log is in real use and the shape of useful extraction queries is clearer from practice.

Relationship to other authoring routes. The disk mirror is a fourth way to modify a score, alongside the JSON tab, the Behaviours tab, and the canvas direct-manipulation toolbar (see Section 14). All four routes converge on the same in-memory bundle, the same explicit-save model, and the same parse-mutate-stringify edit pipeline. AI edits via the mirror end up calling the same bundle.updateContent path that the canvas toolbar calls, so the existing dirty-state tracking, the auto-rerun mechanics, the formatting-preservation logic, and the property inspector's data binding all apply without modification.

Timeline placement. The composition mirror is implemented in stage 4 of the Electron migration (see Section 29). The runtime log is a planned phase scheduled after the composition mirror is stable. Both features are Electron-only; users running the web build (if one continues to ship) do not have access to either, since neither can be made reliable in a browser context.
