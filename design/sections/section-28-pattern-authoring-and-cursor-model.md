# Section 28 — Pattern Authoring and the Cursor Model

Status: design intent for upcoming work. Captures the inspector layout, the cursor-as-collider model, and the visualisation choices that together form the user-facing surface of GXW's strudel integration. Section 27 covers the language and runtime layers (architectural split, pattern evaluation flow, modifier vocabulary, signals, tempo); this section covers how the user authors patterns and how the system visibly enacts them. The two sections evolved out of one combined section after the inspector design began rapidly evolving and the architectural content stayed stable; splitting keeps each file focused on a single concern.

## Object identifiers

This subsection captures the integer-based identifier scheme that replaces the earlier `sp_xxxxxx` / `tr_xxxxxx` / `cv_xxxxxx` hex format. Designed during the Pre-Tier-2 housekeeping pass; supersedes the hex-format text that earlier subsections were written against.

Every object in scene.json carries a system-assigned id with a fixed shape: a three-uppercase-letter prefix denoting the kind followed immediately by a positive integer, with no separator. Sprites are SPR1, SPR2, SPR3; triggers are TRG1, TRG2; curves are CRV1, CRV2. The integer is assigned in creation order per kind and never repeats. A deleted object's number is not reused on the next creation, so deletion leaves gaps in the numbering that remain visible for the lifetime of the score. Name stability is the reasoning: a labelled pattern block $SPR4 in behaviors.js stays meaningful as long as SPR4 exists, and recycling that number for a different freshly-created sprite later would orphan or silently rebind the reference.

The next-integer counters live in scene.json as a top-level `idCounters` sub-object with `sprite`, `trigger`, and `curve` keys, each holding the integer that the next-created object of that kind will receive. Counters monotonically increase. The loader's fill-pass advances each counter past the largest integer it finds in a conventional id, so a hand-edited scene.json that jumps from CRV3 to CRV10 has its curve counter pulled forward to 11. Objects missing an id entirely get one assigned by the fill-pass at the counter's current value, after which the counter increments.

Hand-edited ids that don't match the conventional shape are left alone — the fill-pass treats them like user-supplied identifiers — and the counter doesn't try to reconcile against them. References elsewhere (labelled blocks, callback function names) to such hand-edited ids are the user's responsibility to keep in sync; the system has no opinion. The same applies if the user hand-edits a conventional id to a different value: any existing pattern-block or callback-function references to the original id become orphans, which is fine because we can't prevent the user from doing that.

The conventional shape doubles as the identity regex used to reject user-typed Names that would collide with system-assigned ids. The pattern is `/^(SPR|TRG|CRV)[1-9]\d*$/`: a three-letter prefix from the fixed set followed by a positive integer with no leading zero. The user-typed Name field is currently not exposed in the inspector (the Identity row dropped it during the Tier 3 reshape) but the schema position remains; when Names are re-surfaced their validator will consult this pattern.

The naming scheme flows through to pattern-block labels and callback function names. A labelled pattern block for sprite SPR1 is written `$SPR1: ...` in behaviors.js; the callback functions for the same sprite are `hasHit_SPR1`, `beenHit_SPR1`, and `onTick_SPR1`. The inspector's Pattern row Create button scaffolds `$SPR1: sound("bd sn bd sn")` at the end of behaviors.js — a starter pattern that parses cleanly and fires audibly so the round trip from Create to first audible output lands without a parse-error step in between. The Band 3 row Create buttons scaffold function stubs named `hasHit_SPR1(ctx) { ... }`, and so on. All of these are direct concatenations of the slot name (or dollar prefix) with the object id.

The new-score template seeds the counters one past the highest id assigned in the template. The default template ships with one sprite, four triggers, and one curve, so the template's scene.json has `idCounters: { sprite: 2, trigger: 5, curve: 2 }`.

Transition from the earlier hex-id scheme. Scores authored under the previous `sp_xxxxxx` / `tr_xxxxxx` / `cv_xxxxxx` format load without crashing — the loader accepts any string as an id — but their pattern blocks and callback functions in behaviors.js reference ids that won't match anything in scene.json under the new scheme, so those references go orphan and the scores don't fire patterns. No migration tooling is provided; the existing scores were test artefacts that weren't yet wired to play audio, and users with such scores should start fresh under the new scheme.

## Inspector integration

The strudel pattern integration restructures the property inspector's lower bands. The existing Band 2 (Geometry and visual) is unaffected. Band 1 (Identity) picks up the cycle duration field; Bands 3 through 6 of earlier GXW versions are reshaped, collapsing into a single callback-slots band. The auto-message-interval mechanism is eliminated; autonomous output for any source comes from its cyclePattern when it has cursor extents and is unmuted (per the cursor-as-collider model in the next subsection).

Band 1 carries the cycle duration row in addition to the existing identity row. The cycle duration row reads as "Cycles In [N] beats" with the small numeric field (schema key beatsPerCycle) sitting between a "Cycles In" label on the left and a "beats" units suffix on the right. The field is universal across kinds (curves, sprites, and triggers all have a cycle counter and a wall-clock cycle period), but greys for triggers since triggers cannot self-fire under the cursor-as-collider model and their cycle counter is internal-only. The Identity row drops the user-typed Name field; the system-assigned object id (per the Object identifiers subsection above) is the only identifier the user sees in the inspector. The schema field for name remains in place so existing saved scenes preserve their names without a migration, and a future change can re-surface the field.

Band 3 is the callback-slots band, with four rows: hasHit, beenHit, onTick, and pattern. The first three rows carry a Can-X gate checkbox, a function-name reference field, and a Create or Go-to button that scaffolds a stub in the Code tab when the named function does not yet exist. The fourth row is the pattern row: a row label, no checkbox (cursor presence is derived from cursor extents and mute per the next subsection, so there is no canCycle gate), no function-name field (the object's id is the implicit reference into the Code tab), and a Create or Go-to button that scaffolds an objectId-labelled block in the Code tab when no such block exists or scrolls to the existing block when one does. There is no editor for the pattern inside the inspector; all pattern editing happens in the Code tab. The canCycle gate is gone (replaced by the cursor-extents-and-mute derivation), the beatsPerCycle field has moved to Band 1 as the cycle duration row, and the inline pattern editor that an earlier version of this section described as Band 4 is removed.

The Code tab is the single home for pattern editing. Patterns are expressed as JavaScript labelled statements of the form `$objectId: expression`, where objectId is the system-assigned id of a curve, sprite, or trigger, and the expression is any strudel pattern expression — a function call such as `note("c d e f")` or `s("bd sn")`, optionally chained with modifiers such as `.fast(2)` or `.s("piano")`, optionally wrapped in combinators such as `stack(...)` or `cat(...)`. The labelled-statement syntax is valid JavaScript at the parser level (label, colon, statement), which lets the Code tab keep its existing role as a JavaScript file while picking up the new role as the home for pattern source. The strudel REPL's `$:` convention is exactly this labelled-statement form with an empty label; GXW extends the convention by using the label name to carry the object reference.

The Code tab can contain multiple labelled blocks targeting the same object as variants. The composer types `$SPR1: note("c d e")` once, then later `$SPR1: note("c d e f g")` on another line group, then later `$SPR1: stack(...)` again. All three blocks coexist in the file as a kind of pattern library for that object. Cmd-Enter on the block under the cursor promotes that block to the active pattern for its named object. The previous active block remains in the file as text; the user can switch back to it later by parking the cursor in it and Cmd-Entering again.

The scene runtime cache. Per-object cyclePattern in scene.json stores the currently-active pattern expression for that object as plain text — the expression body alone, with no objectId prefix. Cmd-Enter on a labelled block in the Code tab extracts the block's expression, writes it to that object's cyclePattern field, and re-runs the scene. The scene runtime reads cyclePattern directly when evaluating each cycle. Scene reload from disk uses cyclePattern as-is without re-parsing the Code tab for patterns. The Code tab file is the editing surface; cyclePattern is the runtime cache. This separation means scene reload does not depend on the Code tab containing any patterns at all (a freshly-loaded score with no labelled blocks still runs whatever cyclePattern values are stored in scene.json), and the user can rearrange or comment out blocks in the Code tab without disturbing what is currently playing.

Inertness at scene-load. Behaviors.js historically runs as JavaScript at scene-load time to register the top-level callback functions (hasHit_objectId, beenHit_objectId, onTick_objectId) into scene.functionMap. The labelled pattern blocks are syntactically valid JavaScript labelled statements, so they would also execute at scene-load under the naive path; that is the wrong behaviour, because executing the block calls strudel pattern constructors at load time (which would either error before the engine is loaded or, worse, succeed and produce stray Pattern objects with no live binding). The scene-load path pre-processes behaviors.js through Acorn (already a Code tab dependency for the syntax linter), walks the top-level statements, and splits them into two streams: LabeledStatement nodes whose label matches a dollar-prefixed identifier are pulled out and held as inert source text in a per-object pattern table; everything else is concatenated back into a JavaScript source string and executed as before for function registration. The labelled blocks remain in the file as the user sees them, but they are not run at scene-load. Cmd-Enter is the only path that activates them.

Active-block indication. When one of several labelled blocks for a given object is the currently-active pattern for that object, that block's tag (the `$objectId:` text including the colon) renders in GXW's accent green — the same colour used for editable field borders, checkboxes, and the wordmark glow. Inactive tags render in the default name-token colour, the lighter pink from cmTheme.js's contrastOverrides. The matching rule is text-equality: a block is active for its named object when the block's expression body equals the cyclePattern text stored in that object's scene.json entry. A duplicated block (two blocks for the same object with identical expression text) highlights both as active, which is an honest reflection of state; the rule could be more sophisticated (line-range identity, content-hash plus position) but text-equality is simple and robust to file edits. Implementation is a CodeMirror decoration extension that parses the file, walks for labelled statements, looks up each block's named object in the scene, and adds an active-tag mark when the expression matches; the extension recomputes on doc change automatically and re-triggers via a CodeMirror StateEffect when scene.json's cyclePattern changes.

Bidirectional navigation between Properties and Code tabs. The pattern row's Create / Go-to button in Band 3 is the primary inspector-to-Code-tab path: clicking switches to the Code tab and either scrolls to the object's first occurrence — a labelled pattern block or any callback function whose name follows the slotName_objectId convention — when one exists, or scaffolds an empty labelled block at the end of behaviors.js and drops the cursor there when none does. The first three Band 3 rows' Create / Go-to buttons retain their existing role of scaffolding or navigating to the corresponding callback function. Beyond the explicit Go-to clicks, the two tabs follow the canvas selection while they are active: in the Code tab, when the canvas selection changes to a single object and the Code tab is the active tab, the editor scrolls to that object's first occurrence in the file (same lookup as Go-to). When the canvas selection is multi-object the editor does not scroll — the user is probably working on something that isn't focused on one specific pattern, so disturbing the editor's scroll position would be more disruptive than helpful. In Properties, when the user switches from the Code tab to Properties, the Code tab's cursor position is read and its enclosing block's object id (if any) is checked against the current canvas selection. If the object is already part of the selection, the selection is preserved — the user may have set up a deliberate multi-selection for some purpose, and parking the cursor in one of its members' code reads as navigation within that set rather than a request to abandon it. If the object is not part of the current selection, the selection is replaced with just that one object — the user's focus has moved to something else, so the previous multi-selection is no longer relevant. A cursor on a non-recognised top-level statement or a blank line leaves selection unchanged in either case. Tab switches themselves are explicit user actions — selecting an object on canvas does not auto-switch tabs. The keyboard shortcut for Properties to Code tab switching pairs with the navigation rules so the round-trip from "I am editing SPR1's pattern" to "I am editing SPR1's properties" and back is a key plus the natural follow-the-selection behaviour.

Untagged blocks in the Code tab. A top-level expression with no labelled prefix is just ordinary JavaScript and executes as before at scene-load. Cmd-Enter on an untagged block does nothing pattern-related (it falls through to the Code tab's existing Run Scene gesture). This keeps the Code tab usable for plain helper code (constants, utility functions, scene-load side effects) alongside the labelled pattern blocks. The labelled-statement detection on Cmd-Enter is an additive overload of the existing Cmd-Enter binding, not a replacement: when the cursor is inside a labelled block, Cmd-Enter activates the pattern for that object; when the cursor is anywhere else (inside a function definition, on an untagged top-level statement, on a blank line), Cmd-Enter behaves as it did before.

The function-naming convention applies to the three callback slots (hasHit, beenHit, onTick): the slot name followed by an underscore and the source id. So every source has up to three functions in the Code tab plus zero or more labelled pattern blocks. This makes the Code tab read clearly at a glance: every hit-detection function starts with the hasHit prefix, every collision-response function with beenHit, every per-tick body with onTick, and every pattern block with a dollar-prefix tag. The slot's reference to the function follows the convention by default. Renaming the function in the Code tab away from the convention detaches it from the source slot; the system detects the mismatch on commit, the inspector's slot status indicator goes red, and a soft warning logs to the console naming which source's slot has become orphaned. The user repairs by restoring the function name, by editing the slot to point elsewhere, or by leaving the source silent until they sort it out; the rest of the score keeps playing.

The cycle duration field in Band 1 is a positive number (default four) specifying the source's cycle length in master beats. The master beat is implicitly a quarter note. Cycle duration in seconds is the field times sixty divided by master BPM. Within-cycle event density is expressed by mini-notation modifiers (.fast, .slow, .every, and the rest), so a single cycle-length field is enough to specify cycle timing; sub-beat density does not need its own inspector control because the pattern itself carries it.

Validation has two surfaces. For the pattern blocks in the Code tab, parsing runs only at Cmd-Enter time, not as the user types. On parse failure the editor keeps focus, the GXW console logs the diagnostic in error styling with source identification, and the edit does not propagate to the scene. On parse success the expression body is written to the named object's cyclePattern field, the active-tag highlight moves to the just-Cmd-Entered block, the scene re-runs, and the console logs the parsed event positions. Live parsing on every keystroke was considered and rejected: a partial input mid-type — an unfinished function call, an unclosed string — would flag as broken even though the user simply has not finished typing, which is more disruptive than helpful. The standard Code-tab JavaScript syntax linter (Acorn-driven, debounced) continues to flag syntax errors at the file level via the existing lint-gutter mechanism; the strudel-specific pattern parse only runs on the labelled-statement bodies at Cmd-Enter time. For the function references on hasHit, beenHit, and onTick, the slot's status indicator is green when the named function resolves and runs cleanly, yellow if the function exists but threw the last time it was called, red if the function does not exist; a hover tooltip carries a short message and the full diagnostic is logged to the console. Console messages always include the source object id and slot name so the console is searchable by source even when the inspector is closed. Hard rejection is reserved for pattern parse failures at Cmd-Enter time; soft warnings cover modifier usage known not to fit the source's firing model, the orphaned-function-name case, and the function-throws-on-call case for code-tab bodies. A user who understands the implications can commit a soft-warned pattern; the parse and evaluation are correct, just the typical intent does not match the result.

## Playback control during authoring

Iterative pattern testing demands low-friction control of the transport. The composer types a labelled block, hits Cmd-Enter to promote it, listens for a moment, edits, listens again. Each cycle of edit-and-listen ideally costs no more keyboard or mouse effort than the editing itself. Three transport-toggle gestures plus an engine-load auto-trigger flatten that friction.

Engine-load auto-trigger. The strudel runtime needs a browser user gesture before its AudioContext can resume, and Web MIDI also surfaces its first-time permission prompt only inside a gesture-driven event handler. Both requirements were originally satisfied by a Load Engine button in the transport bar that the user clicked on every page load. The button still exists as a visible affordance, but the runtime now also initialises on the FIRST user interaction of any kind across the document — a click anywhere, a keypress anywhere. The first interaction trips the load and the listeners self-remove via AbortController. In practice the user lands on the page, clicks the canvas or starts typing, and the engine loads behind their first natural action without needing the button at all. The button stays for users who want an explicit affordance and for cases where engine state needs to be visible.

Transport toggle: three gestures, one behaviour. The transport's play and pause states collapse into a single toggle command, fired by any of three independent gestures. Each is just transport.toggle() under the hood; the three exist because each suits a different working position.

Spacebar follows the universal music-app convention. Active globally EXCEPT when focus is in a text-input context — a CodeMirror editor (the Code or Properties tabs), an input or textarea element, or a contenteditable region (the inline score-rename surface). Inside those contexts Spacebar inserts a space character as expected; everywhere else (canvas, inspector buttons, menu bar, message area, body itself), Spacebar toggles the transport. The text-context check is identical to the one the Delete-key handler uses, so the two stay consistent about what counts as "typing".

Cmd-Period is the always-active alternative. Works everywhere including inside the Code tab and Properties JSON tab, so the user can toggle playback without leaving the editor. Cmd is the natural modifier for an audio app on macOS — matches the convention in Logic, GarageBand, and most others. The historical macOS-system Cmd-Period = cancel convention is not strong enough in modern macOS to interfere with deliberate app-level bindings. Ctrl-Period and Ctrl-Space were both considered and rejected: Ctrl-Period reads as the wrong modifier for an audio app, and Ctrl-Space is the conventional autocomplete trigger in CodeMirror and strudel which we expect to bind once autocomplete lands in the Code tab.

Double-click on empty canvas background. The third gesture sits on the canvas itself: a double-click on empty space (not on an object) toggles the transport. Object double-click still emits openObjectInCode for navigation, so the two gestures do not conflict — the canvas hit-test decides which fires. This gives the composer a mouse-only path to play/pause that complements the keyboard paths, useful when one hand is on the trackpad and the other on a coffee cup.

The redundancy across the three gestures is deliberate. The same toggle is reachable from any of the contexts a composer might be in: text-editing in the Code tab uses Cmd-Period; canvas-focused composition uses Spacebar or double-click; menu-bar or inspector interactions use Spacebar. Each gesture is one keystroke or click, so there is no "primary" shortcut to remember — whichever is closest to your current activity is fine.

Implementation note: the keyboard shortcuts live in main.js as window-level keydown handlers, alongside Escape (tool disarm), Cmd-S (save), and Delete (selection remove). Spacebar previously had a duplicate binding in transportBar.js that fired alongside the main.js handler and produced two toggle calls per keystroke (audibly: zero net change because the pair cancelled out); that binding was removed when the main.js handler landed. The R-for-rewind binding stays in transportBar.js as the one transport keyboard shortcut that doesn't overlap with the canvas/editor ergonomics main.js owns. Cmd-Period was never bound in transportBar.js, which is why it worked from the moment its main.js handler landed; Spacebar needed the deduplication step before it landed cleanly.

First-gesture playback gate. The three toggle gestures are all gated on the runtime being fully loaded (status === "loaded"). Without the gate, the same first interaction that triggers the engine-load auto-trigger would ALSO toggle playback, producing surprise audio the moment the engine becomes ready. With the gate, the first gesture is consumed by the engine-load handler; once status flips to "loaded", subsequent gestures toggle normally. The gate is a one-line check at the top of each toggle path.

## Cursors, marker layout, and the cursor-as-collider model

This subsection captures design refinements landed on 2026-05-09 (evening) as Stage 3 of the Tier 3 inspector integration was wrapping up. The earlier subsections describe section 27 as originally written; this subsection updates them where they no longer hold and supersedes them where they conflict.

The canCycle gate is gone, replaced by a two-condition gate combining a derived signal with the existing mute field. A source has a visible, firing cursor iff its cursorL or cursorR field is non-zero AND its mute field is unchecked. The cursor extent fields are presentation parameters that control the cursor's size and shape; mute is the operational toggle that lets the user silence the source's self-firing without losing the extent settings. The user expresses cursor intent by giving the source a cursor extent and toggles the cursor on and off via mute. cursorL and cursorR were always present on curves; sprites now also carry these fields. Triggers do not, since triggers do not move and cannot have cursors. The schema field canCycle is removed. When mute is checked, the cursor extent fields render greyed in the inspector and the cursor disappears from the canvas; the visual matches the absence of audio. Reusing mute conserves vocabulary: the existing mute field already means "this source produces no sound", and today's refinement extends the metaphor by saying that for a source whose audio comes from its own cursor sweeping, no sound implies no cursor visible, since the cursor is the visible cause of the sound. The cyclePattern field is the runtime cache for the currently-active pattern expression on each source; the editing surface is the Code tab via labelled-statement blocks tagged with the object id (see the Inspector integration subsection above for the full Band 1 and Band 3 reshape that ships alongside this model).

The cyclePattern field is the source of truth for two things: the audio firing pattern (when the source has a cursor) and the visual marker layout along the source's path (always, on curves; deferred for sprites). The two interpretations agree on positions, since both sample the same parsed Pattern via queryArc against the same cycleDuration. A curve with cyclePattern "x . x ." and cursorL=cursorR=0 shows two diamond markers at fractional positions 0 and 0.5 along its geometry and fires nothing on its own. The same curve with non-zero cursor extents and mute unchecked additionally shows a cursor that sweeps the geometry once per cycle and fires audio events as it crosses each marker. The marker rendering is unconditional; the audio firing is gated by the combination of cursor extents being non-zero and mute being unchecked.

The lockstep illusion. Because the cursor's t advances at one full cycle per cycleDuration and the pattern's events are scheduled at fractional cycle times multiplied by the same cycleDuration, the cursor reaches each marker position at exactly the moment that marker's audio event fires. The user reads this as cursor-causes-firing because that is the simplest causal story their eye can construct, even though under the hood the pattern is firing on its own scheduler and the cursor is independently traversing the geometry. They share one clock, so they are always in geometric agreement. For line and ellipse curves the t-parameterisation is uniform along arc length, so the visual lockstep is exact in screen coordinates. For piste curves with non-uniform parameterisation the lockstep is exact in t-space; whether arc-length non-uniformity is visually perceptible is something to find out in practice.

Marker re-rendering on cycle wrap. Markers re-render at the start of each cycle by re-querying the curve's parsed cyclePattern. For deterministic patterns the positions are stable cycle to cycle and the re-render has no visible effect. For stochastic patterns (using degradeBy, choose, scramble, the rand or irand signals) the positions shift on each wrap and the diamonds visibly animate. When the curve also has a cursor and is firing audio, the cursor's traversal is in lockstep with whichever positions the current cycle produced, so the visual coincidence holds even for stochastic patterns.

Free-sprite cursors. Sprites now carry cursorL and cursorR fields, mirroring curves. A free sprite's cursor is a line segment perpendicular to the sprite's last motion direction, with cursorL units extending one side and cursorR units the other. Between consecutive simulation ticks the cursor sweeps a quadrilateral that triggers (and possibly other sprites) are tested against. When the sprite is stationary the cursor persists in its last non-zero motion direction, so the cursor is visually anchored when the sprite slows to a stop; a sprite that has never moved gets an arbitrary default direction (horizontal), but that case is unusual and visible to the user. cyclePattern firing for sprites happens at the sprite's current position when each event's fractional cycle time arrives, rather than at fixed positions along a path; sprites have no fixed path so the marker layout interpretation does not apply. The audio firing path is the only operational meaning of cyclePattern for sprites.

Cursor-as-collider. A sprite without a cursor cannot collide with anything. Sprites are conceptually point objects, so collision between two points is degenerate without some swept area. Requiring a cursor for collision dissolves this problem cleanly: a user who wants a colliding sprite gives it a cursor extent; a user who wants a passive moving point leaves the extents at zero. The same gesture controls visual presence and collision capability, which lines up the user's authoring action with both the visible result and the operational behaviour. Cursors on curves work the same way: a curve without a cursor is a passive geometry and a marker container; a curve with a cursor is a collider and an audio source.

Derivation extends to canHit. By the same logic the canHit gate (whether a source can hit other sources) collapses into the cursor-presence derivation, since a source can only do the hitting via its sweeping cursor. canBeHit stays a separate explicit gate, since being hittable does not require a cursor: anything with a body (curve geometry, sprite point, trigger position) can be hit by another's cursor. canTick stays a separate gate too, since onTick is purely time-based and has no geometry involvement. The schema field canHit may follow canCycle into removal in a future commit; the field stays for now and the derivation lands when collision firing is wired.

Deferred. Marker layout on sprites by predicting their trajectory is interesting (the sprite's near-term path could be sampled forward and markers placed along the predicted positions) but gets complicated fast: collisions perturb the prediction, and a marker the sprite was supposed to hit could be invalidated by an intervening wall bounce. Keeping the simple invariant "markers belong to curves" for now. Sprite-vs-sprite collision is also deferred; sprite-vs-trigger covers most of the use case and is geometrically simpler.

What happens to the audio when a curve has cyclePattern but no cursor. The pattern's events do not fire on the curve's own clock. The markers are still rendered along the geometry. When another cursor (from a different curve, or from a free sprite) sweeps over those markers, the most natural firing is: the OTHER source's hasHit fires (it hit something), THIS curve's beenHit fires (it got hit at marker N), and beenHit's ctx carries the Hap payload from the pattern (sound name, gain, whatever the pattern scheduled for that event). This unifies the cursor-driven and marker-collision interpretations: the pattern always describes the rhythmic content; whether the source fires it itself or has it fired by an external cursor is the user's choice, expressed by giving or not giving the source a cursor. The detailed mechanism for ctx population on a marker-collision-driven beenHit is deferred until collision firing is wired; flagging the shape because it falls out cleanly from the cursor-as-collider model.

Mute and external collisions. Mute silences only the source's own self-firing: the audio that fires when the source's cursor sweeps over its own markers. External collisions with the source's markers fire normally regardless of mute state, since those collisions are credited to the colliding source's hasHit callback (its own audio output) and to the muted source's beenHit callback (whose own gate canBeHit is independent of mute). This keeps the passive-curve-as-trigger-field use case clean: a curve laid out with cyclePattern and muted so it produces no self-firing remains audible when other cursors sweep across it, with the sound credited to those cursors and to the curve's beenHit binding rather than to the curve's own muted output. For triggers, mute is a no-op behaviourally. Triggers cannot self-fire under the cursor-as-collider model (they have no cursor), so there is nothing for mute to silence; the field stays in the schema for uniformity with curves and sprites, and a muted trigger renders the same way an unmuted one does.

## Visualisation

The existing beat-point glyphs on curves assume evenly-spaced active beats and don't extend cleanly to arbitrary mini-notation patterns. Two options for strudel mode visualisation are under consideration; neither is chosen yet.

Static glyphs at edit time would query the pattern for one cycle when the user commits a pattern edit and place visual markers on the curve at the resulting fractional positions. Works cleanly for deterministic patterns; breaks down for stochastic ones (like patterns using degradeBy or randomly-sampled choice operators) where each cycle's events differ.

After-the-fact breadcrumbs would drop a fading marker at each event's actual played position when the audio plays, leaving a visible trail of recent events. Works for any pattern. Doesn't pre-show where the next firing will be, but for a music tool where the user is composing in real time and listening, the retrospective view is often the more useful one anyway.

A hybrid is plausible — static glyphs at edit time for deterministic patterns, after-the-fact breadcrumbs for stochastic ones, with detection of which mode applies happening at parse time. Resolution deferred until the rest of the integration is implemented and the visual surface gets focused attention.
